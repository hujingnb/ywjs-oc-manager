## 上线阻断

### 1. 容量门禁失败

100 并发、30 分钟正式测试结果：

- 总请求 `77,994`，成功 `994`，成功率 `1.274%`，远低于 `99.5%` 门槛；
- HTTP 429：`26,688`，HTTP 502：`68`，网络错误：`49,459`，超时：`785`；
- 会话串写为 `0`，但该单项无法抵消成功率与可用性失败；
- 第二次按虚拟访客隔离来源 IP 后结果仍失败，说明不是负载工具共享来源地址造成的假象。

状态：`AICC-LOAD-01 = FAIL`。

### 2. 最终故障演练缺少健康模型基线

已验证 Hermes 和 manager-api 重启后的会话恢复与消息幂等；但在依赖演练复跑期间，即使重启全新 Hermes、RAGFlow/new-api Pod 均 Ready，首条健康基线消息仍在 30 秒后返回：

```text
HTTP 502 CONVERSATION_CLI_ERROR
转发 AICC 消息失败: 会话上游调用失败
```

Hermes 日志同时记录 `deepseek-chat` 上游 180 秒无数据及 `APITimeoutError`。因此无法证明最终环境在健康依赖下可稳定回答，也无法完成 RAGFlow/new-api/Redis/MySQL 的同一轮连续恢复门禁。

状态：`AICC-FAULT-01 = BLOCKED`，`AICC-FAULT-02 = BLOCKED`。

### 3. 最终迁移尚未完成完整升级复验

旧基线升级演练通过到迁移 31；故障测试发现重复消息后新增迁移 32（公开消息幂等键）。迁移 32 已在当前本地数据上成功执行，单元和构建通过，但尚未重新执行完整 `master -> 当前 HEAD -> rollback -> 当前 HEAD` 浏览器演练。

状态：`AICC-UPGRADE-01 = BLOCKED`，`AICC-ROLLBACK-01 = BLOCKED`。

## 已修复问题

- Redis、MySQL、new-api 故障不再向访客暴露原始连接错误；
- 公开消息新增 `client_message_id`，运行时恢复重试不会重复写入访客消息；
- 升级演练补齐基础镜像、manager、ops、Hermes 镜像重建与回滚恢复；
- 故障脚本处理 manager 滚动期间的非 JSON 网关响应，并区分安全降级回复与稳定 5xx 错误。

2. 恢复稳定模型上游，在健康基线可连续回答后完成全部依赖故障注入和恢复。
3. 基于包含迁移 32 的最终提交重新执行完整升级/回滚与两轮浏览器冒烟。
4. 在同一最终提交、同一镜像和干净数据上复跑全部 Playwright，并更新需求矩阵；此前不得将本报告改为 GO。
