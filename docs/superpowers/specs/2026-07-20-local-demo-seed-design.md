# 本地演示数据自动初始化设计

## 1. 背景与目标

当前 `make local-up` 会拉起本地 k3d 全栈、创建平台管理员，并初始化 new-api 与
RAGFlow 模型配置；`make local-reset` 清空 `.k3d-data` 后复用 `local-up` 完成重建。
环境启动后仍需人工创建助手版本、企业、成员、普通实例和智能客服，导致本地联调准备
耗时且不同开发者的数据形态不一致。

本改动让 `make local-up` 和间接调用它的 `make local-reset` 自动补齐固定演示数据。
命令只有在普通实例与智能客服运行时真实就绪后才成功退出。重复执行采用“只补缺、
不覆盖”的幂等语义，保留开发者对既有数据的修改。

本设计仅面向本地 k3d 环境，不用于 E2E fixture、线上或共享环境。

## 2. 总体方案

新增宿主机脚本 `scripts/local-seed-demo.py`，通过 manager 正式 HTTP API 编排初始化，
不直接写 manager 数据库，也不复用 `cmd/seed-e2e`。新增内部 Make 目标
`local-seed-demo`，由 `local-up` 在以下步骤完成后调用：

1. manager、new-api、RAGFlow 和本地存储组件就绪；
2. `seed-admin` 已创建平台管理员；
3. `local-init-models` 已完成 new-api/RAGFlow 配置并回填 manager Secret；
4. manager-api 使用新 Secret 重启并恢复健康。

脚本先以本地平台管理员 `admin/admin123` 登录，补齐助手版本、企业、企业配置和智能
客服。成员生命周期不允许平台管理员写入，因此脚本在确需创建成员时再以对应企业管理员
登录，调用成员创建或 onboarding 接口；已有成员的实例可由平台管理员通过正式复建接口
补齐。最后统一轮询异步任务与资源状态。`local-reset` 保持现有“清空后调用
`local-up`”结构，因此无需再维护另一套初始化路径。

采用 HTTP API 的原因如下：

- 复用正式权限、参数校验、new-api 企业开户、事务、异步 Job 和 Kubernetes 编排链路；
- 避免裸 SQL 把实例直接标记为 `running`，却没有真实运行时资源；
- 避免新增 Go CLI 后复制 `cmd/server` 的依赖装配并产生长期漂移。

## 3. 固定演示数据

### 3.1 助手版本

创建两个全局版本，均使用 `deepseek-chat` 和 runtime image 查询接口返回的第一个合法
Hermes image ID，不硬编码镜像仓库地址。AICC 常驻容器仍使用现有
`aicc.runtime_image` 专属配置；助手版本的 image ID 用于满足版本校验及保存行为配置：

| 名称 | 用途 |
|---|---|
| 本地通用助手版 | 普通成员实例 |
| 本地智能客服版 | AICC 隐藏实例 |

首次创建时写入与用途匹配的简洁中文描述和 system prompt。版本已存在时不更新模型、
镜像、提示词、routing 或其他字段。

### 3.2 企业与资源矩阵

| 企业标识 | 展示名 | 版本 allowlist | 普通实例 | 智能客服 |
|---|---|---|---|---|
| `demo-full` | 完整演示企业 | 智能客服版、通用助手版（按此顺序） | 1 个 | 1 个 |
| `demo-app` | 普通实例演示企业 | 本地通用助手版 | 1 个 | 无预置客服 |
| `demo-aicc` | 智能客服演示企业 | 本地智能客服版 | 无普通实例 | 1 个 |

每家企业首次创建时生成以下 manager 账号：

| 角色 | 账号 | 默认密码 | 显示名 |
|---|---|---|---|
| 企业管理员 | `admin` | `admin123` | 演示管理员 |
| 普通成员 | `member` | `member123` | 演示成员 |

普通实例由 `member` 持有，首次创建名称为“演示助手”。智能客服首次创建名称为
“演示智能客服”，创建完成后自动启动。`demo-full` 和 `demo-aicc` 使用相互独立的
智能客服及隐藏实例。

`demo-full` 与 `demo-aicc` 开启 AICC，agent limit 至少为 1；`demo-app` 首次创建时
保持 AICC 关闭。若复用数据时 `demo-app` 已由开发者开启 AICC，脚本不将其关闭。

## 4. 数据流与幂等语义

### 4.1 预检

脚本首先检查 `.env` 是否同时提供 `DEEPSEEK_API_KEY` 与
`SILICONFLOW_API_KEY`。缺少任一配置立即失败并指出配置键名，不输出配置值。随后等待
manager 健康检查通过并登录平台管理员。

脚本通过 runtime image 与现有资源查询接口建立当前状态，不把本次进程内缓存当作事实
来源。每一个写步骤完成后都使用查询结果确认目标状态。

### 4.2 助手版本与企业

助手版本以固定名称识别，不存在时创建，存在时原样复用。企业以不可变的企业标识
`code` 识别；同 code 的对象类型或关键归属不符合预期时，脚本报冲突并停止。

新企业通过正式创建接口生成 manager 记录、new-api 企业用户和首个企业管理员。
已有企业只做以下单向补齐：

- 保留现有助手版本 allowlist，并追加该场景缺少的固定版本；
- AICC 场景保证能力已开启，并保证有限额度不低于 1；
- 不降低已有 AICC 限额，不移除额外版本，不关闭已有能力；
- 不覆盖企业名称、联系人、配额、备注或管理员密码。

现有 AICC 隐藏实例创建逻辑固定选择企业 allowlist 的第一个版本，创建请求本身不能显式
传版本。因此新建 `demo-full` 时必须按“本地智能客服版、本地通用助手版”写入 allowlist，
普通实例仍显式绑定通用助手版。复用已有企业时不擅自重排 allowlist：若演示客服已存在，
直接复用其既有版本；若需要新建客服但 allowlist 首项不是本地智能客服版，则视为配置冲突
并给出修复提示，而不是静默创建一个行为配置错误的客服。

### 4.3 成员与普通实例

成员以企业内用户名 `member` 识别。账号不存在时用默认凭据创建；已存在时不重置密码、
显示名、角色或状态。

创建缺失成员前，脚本使用该企业 `admin/admin123` 登录，以遵守现有“平台管理员只观察
成员、企业管理员管理成员生命周期”的权限边界。全新初始化时该密码由企业创建请求确定。
复用数据时若企业管理员密码已被修改且 `member` 又恰好缺失，脚本不得重置管理员密码或
绕过权限写数据库，而应报告该企业无法完成成员补齐并给出人工处理提示。若 `member` 已
存在，则无需企业管理员登录；平台管理员仍可通过正式接口为该成员补建缺失实例。

`demo-full` 与 `demo-app` 检查该成员是否已有未删除的普通实例。没有时通过正式
onboarding/成员实例接口创建“演示助手”，绑定“本地通用助手版”；已有实例时直接复用，
即使实例已被开发者改名也不重复创建。若已有资源处于不可恢复的归属冲突状态，脚本失败
并给出企业、成员和实例标识。

普通实例的成功条件是 `runtime_phase=ready`，且业务状态为 `binding_waiting` 或
`running`。微信未扫码只影响渠道绑定，不应被误判为运行时初始化失败。

### 4.4 智能客服

`demo-full` 与 `demo-aicc` 按以下顺序识别演示客服：

1. 优先复用名为“演示智能客服”的未删除客服；
2. 固定名称不存在但企业只有一个未删除客服时，视为开发者改名后继续复用；
3. 存在多个客服且没有固定名称时，因无法安全判断归属而报冲突；
4. 没有客服时，通过正式 API 创建；先等待其隐藏实例运行时就绪，再调用启动接口。

智能客服的成功条件是 agent 状态为 `active`，且关联的 AICC 隐藏实例
`runtime_phase=ready`。脚本不得以 agent 主记录已创建代替运行时就绪判断。

### 4.5 不确定写结果

查询请求遇到连接拒绝、HTTP 429、502、503 或 504 时可做有限指数退避。创建请求若因
连接中断而无法确认结果，不得直接盲目重发；脚本先按稳定身份重新查询，确认对象不存在后
才再次创建，避免响应丢失造成重复数据。

中途失败不回滚已经成功创建的演示对象。下次执行从实际状态继续补齐；单次正式 service
调用内部的跨系统补偿仍由现有业务实现负责。

## 5. 错误处理与输出

以下任一情况使 `local-seed-demo`、`local-up` 和间接调用的 `local-reset` 非零退出：

- 模型厂商 Key 缺失；
- manager 登录或 API 调用失败；
- 固定身份与现有资源发生无法安全解析的冲突；
- Job 进入失败终态；
- 普通实例或智能客服未在 15 分钟内达到成功条件。

API 错误输出包含操作名称、目标企业/对象、稳定错误码和安全消息。Job 错误额外输出
Job ID、失败阶段和错误原因。日志不得包含厂商 Key、access token、refresh token、
new-api 凭据、AICC public/widget token 或其他秘密。

成功时输出两个助手版本、三个企业、两个普通实例和两个智能客服的简短状态汇总，并保留
manager、new-api、RAGFlow 的本地访问地址提示。

## 6. 文档同步

更新根目录 `CLAUDE.md` 的“本地调试账号”章节，增加三个 manager 企业的登录信息：

- 企业标识 `demo-full`、账号 `admin`/`member`；
- 企业标识 `demo-app`、账号 `admin`/`member`；
- 企业标识 `demo-aicc`、账号 `admin`/`member`。

文档明确这些是 `local-reset` 后首次创建的默认密码。复用已有 `.k3d-data` 执行
`local-up` 不会重置开发者修改过的密码。上游内部账号、随机 token 和厂商 Key 不写入
文档。

同步更新 `docs/local-development.md` 的 `local-up`/`local-reset` 行为和初始化失败处理，
避免命令说明仍声称只创建平台管理员。

## 7. 测试与验收

### 7.1 自动化测试

为 Python 编排逻辑增加标准库 `unittest`，使用可控的假 HTTP 依赖覆盖：

- 空环境首次创建全部固定数据；
- 第二次执行不产生重复版本、企业、成员、实例或客服；
- 人工修改实例/客服名称和账号密码后不被覆盖；
- allowlist、AICC enable 和 agent limit 只做单向补齐；
- 创建响应中断后先查询再决定是否重试；
- 缺少厂商 Key、资源冲突、Job 失败和等待超时均返回失败。

每个测试方法、子测试和 table-driven 数据均按仓库规范添加相邻中文场景注释。执行
Python 定向测试，并用 `make -n local-up` 检查 Make 调用顺序。

### 7.2 本地集成与浏览器验收

1. 真实执行 `make local-reset`，确认全栈与演示运行时全部就绪；
2. 再执行一次 `make local-up`，确认没有重复数据且人工可修改字段不被覆盖；
3. 使用真实无头浏览器进行定向验证：
   - 平台管理员可看到两个版本和三个企业；
   - 三个企业管理员均可使用文档中的默认凭据登录；
   - `demo-full` 同时显示普通实例和智能客服；
   - `demo-app` 显示普通实例且没有预置客服；
   - `demo-aicc` 显示智能客服且没有普通实例；
   - 两个预置智能客服的公开页面均能完成一轮真实问答；
4. 检查 `CLAUDE.md` 账号表与实际创建结果一致；
5. 检查工作区没有混入无关改动、秘密或临时调试文件。

全量 E2E 不作为本改动的默认验证；浏览器验收仅运行覆盖本地初始化结果的定向场景。

## 8. 非目标

- 不把演示种子能力用于生产或共享集群；
- 不把本地演示数据合并进 Playwright `seed-e2e` fixture；
- 不创建行业知识库、上传示例文档、预置会话或线索；
- 不为演示数据增加通用配置 DSL、可变数量或外部模板；
- 不修改业务 API 契约，也不手工编辑 OpenAPI 生成产物。
