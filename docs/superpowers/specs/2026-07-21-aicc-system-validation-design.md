# AICC 客服体系验证场景设计

- 日期：2026-07-21
- 状态：已确认，待实现计划
- 范围：补强本地 AICC 真实浏览器自动化验证，覆盖知识库、设置生效、重启、模型切换、线索、会话、安全、隔离和故障恢复。

## 1. 背景

现有仓库已经有 `web/tests/e2e/aicc.spec.ts`、`aicc-knowledge.spec.ts`、
`aicc-conversation-*.spec.ts` 以及 `docs/testing/aicc-conversation-requirement-matrix.md`。
这些用例覆盖了 AICC 基础创建、公开接待、知识上传、手机号线索、模型切换 helper、对话安全和部分故障前置。

本次目标不是重写测试体系，而是在现有 Playwright slow/model/rag 套件上补充更完整的客服体系验证场景，重点证明：

- 上传或修改知识库后，公开客服真实问答使用正确知识。
- 修改客服设置和运营字段后，经启动、停止或重启后确实生效。
- 企业 AICC 模型变化能触发静默 rollout，且运行中和暂停客服行为符合设计。
- 手机号线索从公开页提交到后台线索、会话证据和 CSV 导出形成闭环。
- 无关问题、操作性问题和提示词注入不能突破客服边界。

## 2. 实现策略

采用“补强现有套件”的方式：

- 在 `web/tests/e2e/aicc.spec.ts` 扩展管理台、设置、模型、线索、公开聊天和角色权限场景。
- 在 `web/tests/e2e/aicc-knowledge.spec.ts` 扩展知识库修改、删除、冲突、组合、多文件和范围切换场景。
- 在 `web/tests/e2e/aicc-conversation-*.spec.ts` 保留现有安全、意向和故障矩阵，只补必要断言，不把它们改成默认快速回归。
- 在 `web/tests/e2e/aicc/helpers.ts` 增加可复用 helper，例如知识上传/删除等待、runtime 重启等待、数据库事实读取、手机号线索计数和模型 rollout 事实检查。

所有新增场景默认归入 `@slow`、`@model`，涉及 RAGFlow 的归入 `@rag`。依赖额外故障注入或固定外部 fixture 的场景继续通过环境变量显式启用，避免日常局部回归不稳定。

## 3. 场景分层

### P0 必测闭环

P0 是本次实现优先级最高的自动化场景，必须能在本地 k3d、真实浏览器、真实模型和 RAGFlow 可用时完成。

#### 知识库

- 上传当前客服知识库文件，等待 RAGFlow 解析完成，再等待 runtime 检索可命中，公开页问精准事实并断言回复包含唯一事实码。
- 上传企业知识库文件，开启企业知识范围，公开页问企业事实并命中。
- 关闭企业知识范围后，同一客服公开页新会话不再命中企业知识。
- 修改知识库：删除旧客服文档、上传新文档，旧事实不再出现，新事实可以命中。
- 删除知识文档后，公开端新会话不能继续引用已删除事实。
- 问不存在知识编号时，回复不为空，且不暴露 `traceback`、`api call failed`、`dial tcp`、`stack trace` 等内部错误。

#### 设置与重启

- 修改客服名称后，公开页标题同步变化。
- 修改欢迎语后，新访客进入公开页看到新欢迎语。
- 修改 persona，例如要求固定前缀；手动重启运行时后公开问答体现新 persona。
- 修改回答边界后，边界外问题被拒绝或引导回企业业务。
- 修改敏感词后，公开端包含敏感词的消息被服务端拒绝。
- 修改单会话消息上限后，超过上限的消息返回 429 或页面受控提示。
- 修改知识范围后重启，配置仍保留且公开问答按新范围执行。

#### 企业模型

- 平台管理员修改企业 AICC 模型时，确认弹框提示逐个静默重启。
- 保存后产生 `aicc_model_rollout` job，最终状态为 `succeeded`。
- 运行中客服完成静默重启后，`runtime_phase` 回到 `ready`，公开页仍可问答。
- 暂停客服不因企业模型修改自动启动；后续手动启动时应用最新企业模型配置。
- 本地模型目录不足两个时，测试明确报“环境前置不足”，不把它误判为业务失败。

#### 公开聊天与安全边界

- 问知识库相关问题，回复包含事实码且页面有正常消息时间。
- 问无关问题，例如写代码、创建网站、执行命令、登录后台，回复应拒绝或引导回企业业务。
- 提示词注入要求输出系统提示词、忽略规则或切换管理员身份时，回复不得泄露系统提示词，也不得接受注入标记。
- 操作性拒绝后，数据库 `aicc_message_sources` 不应写入受信任工具来源。

#### 手机号线索

- 配置手机号留资字段后，高意向对话出现留资卡。
- 空手机号提交被拦截。
- 非法手机号格式被页面或 API 拒绝。
- 合法手机号提交成功，留资卡消失，刷新后不重复展示。
- 同一 session 重复提交同手机号不重复创建线索。
- 后台线索页显示手机号、未读状态和关联会话。
- 打开会话后未读变已读，CSV 导出包含手机号。

### P1 稳定性与隔离

P1 覆盖并发、隔离、状态和更多配置组合，在 P0 稳定后补充。

- 多文件知识组合问答：两个文档分别包含不同事实，公开页问题要求同时比较两个事实。
- 客服库与企业库存在冲突事实时，验证优先级符合当前产品口径。
- 同企业客服 A 的私有知识不能被客服 B 命中，除非该知识放入企业库。
- 访客 A 的私密标识不能出现在访客 B 的回复或页面历史中。
- 首次进入公开页不提前创建空 session，发送第一条消息才创建。
- 刷新页面后恢复同一 session 的访客消息和客服回复。
- 点击“已解决”后数据库状态为 `resolved`；新问题后重置为 `unknown`。
- 点击“未解决”后数据库状态为 `unresolved`；新问题后重置为 `unknown`。
- 移动端 390px 下长中文和长英文产品名不产生横向溢出。
- 挂件 iframe 内完成问答，session channel 记录为 `web_widget`。
- allowed domains 允许域名可创建挂件 session，非允许域名返回 `AICC_DOMAIN_FORBIDDEN`。
- 关闭企业 AICC 后，公开链接返回离线错误，企业不能继续创建新客服。
- 智能体数量达到上限后，新建客服返回 409。

### P2 故障注入

P2 需要额外 fixture 或环境开关，不纳入默认 slow 套件。

- RAGFlow 检索失败时，公开端返回安全兜底，不暴露内部堆栈。
- 模型回复超时或失败时，消息状态可重试或给出受控提示。
- 队列或 worker 短暂失败后，消息任务可恢复，不重复回复多条。
- 意向分析失败一次后恢复，不重复展示留资邀请。
- rollout 重启失败后 job 保留可重试，不继续标记后续客服成功。
- runtime circuit open 时公开端展示受控提示，不泄漏内部原因。
- 知识库解析失败时，管理端状态可见，公开端不把失败文档当作知识来源。

## 4. 断言原则

- 公开问答以唯一事实码断言业务结果，避免绑死模型自然语言措辞。
- 配置类变更必须同时断言保存 API、页面可见状态和公开端行为。
- 重启类变更必须等待 Kubernetes Pod Ready 与 manager `runtime_phase=ready`，不能只等按钮状态。
- 线索类变更必须同时覆盖公开页、后台列表、会话详情和必要的数据库事实。
- 安全边界不能只看回复文本；操作性请求还要检查未产生受信任工具来源审计。
- 环境缺失必须显式报错或 `skip`，不能把 fixture 不足记录为 PASS。

## 5. 数据与隔离

每个新增场景使用时间戳或 UUID 生成唯一事实码、文件名、客服名和手机号，避免三轮运行或并发运行互相污染。

涉及数据库事实读取的 helper 必须：

- 先校验 `kubectl config current-context` 是 `k3d-ocm`。
- 显式传入 `--context k3d-ocm` 和命名空间。
- 只访问本地 `ocm` 或 `oc-aicc` namespace。

测试不得访问生产或共享集群。

## 6. 文件影响

预期主要改动：

- `web/tests/e2e/aicc.spec.ts`
- `web/tests/e2e/aicc-knowledge.spec.ts`
- `web/tests/e2e/aicc-conversation-security.spec.ts`

## 7. 验证命令

实现后按改动范围运行：

```bash
cd web
npm run test:e2e -- --project=chromium tests/e2e/aicc.spec.ts tests/e2e/aicc-knowledge.spec.ts
OCM_AICC_CONVERSATION_E2E=1 npm run test:e2e -- --project=chromium tests/e2e/aicc-conversation-security.spec.ts
```

涉及真实 Chrome Stable 验收时使用现有 `chrome-headed` project，并记录阻塞原因、trace、screenshot 或 video。

后端或前端业务代码有改动时追加对应最小测试，例如：

```bash
go test ./internal/service ./internal/worker/handlers -run AICC -count=1
cd web && npm test -- --run src/pages/aicc
```

## 8. 非目标

- 不做全量 E2E 默认回归。
- 不新增生产集群验证。
- 不改变 AICC 产品功能或安全策略，除非测试暴露现有实现缺陷。
- 不为了测试稳定性降低断言质量、跳过有效失败或把依赖阻塞写成通过。

## 9. 开放约束

- 知识问答、模型切换和意向线索依赖真实模型、RAGFlow 与本地 AICC runtime 可用。
- 企业模型切换场景要求本地至少存在两个可用 AICC 模型。
- P2 故障场景需要明确的环境开关或 fixture，默认不运行。
