# 设计文档：项目全量 Bug 排查

**日期：** 2026-05-25
**状态：** 待批准

## 背景

当前目标是检查 `oc-manager` 项目是否存在 bug。仓库是 Go/Gin 后端、
Vue/Vite 前端、PostgreSQL/Redis 依赖和 Runtime Agent 组成的管理平台，最近提交
集中在 job 查询权限、worker 状态推进、Runtime 操作和前端实例详情页。

本次工作先做排查与验证，不直接修复代码。若发现已确认 bug，再为具体修复单独进入
后续设计、计划和实现流程。

## 设计目标

- 建立当前项目的自动化健康基线，包括后端、前端、OpenAPI 同步和构建检查。
- 审计近期高风险变更路径，重点覆盖权限、job/worker 状态、Runtime 操作、
  new-api 余额与用量实时查询约束。
- 使用真实浏览器验证核心用户流程，不能用 curl 替代前端行为验证。
- 输出按严重级别排序的 bug 报告，区分“已确认 bug”“疑似风险”和“无法验证项”。
- 保持只读排查，不修改业务代码、生成产物或测试，除非用户后续明确进入修复阶段。

## 排查策略

推荐采用全链路分层排查：

1. 自动化基线先行：先运行现有测试、类型检查、构建和 OpenAPI 校验，快速发现明确回归。
2. 代码审计跟进：围绕最近提交和项目硬性约束做重点阅读，确认自动化检查未覆盖的逻辑风险。
3. 浏览器验证收尾：启动本地环境后用真实浏览器走关键流程，验证前后端联动和 UI 状态反馈。

这个顺序能先用工具拿到客观信号，再把人工审计集中在最可能出问题的路径，最后用浏览器验证用户可见行为。

## 自动化检查范围

后端检查：

- `go test ./...` 或项目 Makefile 中等价的 Go 单元测试入口。
- `go vet ./...` 或项目 Makefile 中等价入口。
- 重点关注失败测试、panic、数据竞争迹象、权限断言缺口和状态机边界。

前端检查：

- `npm run typecheck`。
- `npm test -- --run`。
- `npm run build`。
- 重点关注类型漂移、接口类型不匹配、查询错误处理和页面构建失败。

契约检查：

- `make openapi-check`。
- 如果该命令因为环境依赖无法运行，记录具体原因和风险，不手工编辑生成产物。

## 代码审计范围

近期提交相关路径：

- job 查询权限和组织隔离。
- `useJobQuery`、任务进度轮询和 4xx/5xx 错误处理。
- app 初始化、重启、运行状态推进和 worker 成功/失败路径。
- Runtime Agent 交互、容器启停、文件 API 代理和健康探针。

项目硬性约束：

- 角色 / 资源权限判断必须集中在 `internal/auth/authorizer.go`。
- 余额和用量必须实时查询 new-api，manager 不缓存、不落库快照。
- `users.deleted_at` 仅表示用户下线时间，不能混同组织真删除语义。
- handler 路由、请求体或响应变化必须保持 OpenAPI 与前端类型同步。

## 浏览器验证范围

本地服务可启动时，使用真实浏览器验证以下关键流程：

- 管理员登录。
- 组织、成员和应用相关的列表/详情页能正常加载。
- 实例详情页中的 Runtime 操作反馈、job 进度展示和错误状态。
- 余额/用量页面是否展示实时数据或清晰错误，不出现缓存快照假象。
- 权限边界：平台管理员、组织管理员和组织成员可见内容是否符合角色预期。

如果本地依赖、镜像或外部 new-api 不可用，记录阻塞点，并尽量用现有测试或代码路径补充证据。

## Bug 判定标准

已确认 bug 需要满足至少一项：

- 自动化检查稳定失败，且失败与现有代码行为有关。
- 浏览器流程可稳定复现错误行为。
- 代码审计发现明确违反项目约束或权限/数据隔离规则，并能指出具体路径。

疑似风险包括：

- 需要特定外部服务或生产数据才能确认的问题。
- 自动化未覆盖但存在明显边界缺口的逻辑。
- 行为与文档或注释不一致，但未能在本地完整复现。

## 输出格式

最终报告按严重级别排序：

- `P0`：数据泄露、权限绕过、破坏性数据写入、核心流程完全不可用。
- `P1`：主要业务流程失败、状态严重不一致、用户可见错误。
- `P2`：边界条件错误、错误提示或重试策略不合理、局部功能异常。
- `P3`：低风险一致性问题、测试缺口、文档与行为轻微不一致。

每条问题包含：

- 标题和严重级别。
- 证据来源：命令输出、浏览器复现步骤或代码位置。
- 影响范围。
- 建议下一步：修复、补测试、需要外部环境确认或暂不处理。

## 不在本次范围

- 不直接修改业务代码、测试、生成产物或配置。
- 不做无关重构、格式化或依赖升级。
- 不创建新功能，不改变权限模型或数据库 schema。
- 不提交 bug 修复；修复需要用户确认具体问题后另走设计与计划。

## 验收标准

- 完成自动化基线、代码审计和浏览器验证三类检查，或明确说明无法运行的原因。
- 输出结构化 bug 报告，所有结论都有证据或明确标为疑似风险。
- 工作区不混入无关改动。
- 若发现 bug，给出下一步修复建议，但不在本次排查中直接实现。
