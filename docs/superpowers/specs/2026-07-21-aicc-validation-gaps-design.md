# AICC 验证缺口补齐设计

## 背景

上一轮客服体系验证补强后，仍有三个明确缺口没有标记为通过：手机号格式校验、暂停客服手动启动后应用最新企业模型配置、无关问题拒绝稳定性。本设计只补这三个缺口，不重构 AICC 对话架构，也不扩大到全量评测体系。

## 目标

- 访客提交 `phone` 类型留资字段时，前端和后端都拒绝明显非法手机号；合法手机号仍按现有联系方式哈希归并正式线索。
- 暂停中的客服遇到企业模型配置 revision 推进后，手动启动必须应用最新 revision；启动后公开接待不再停留在旧模型配置。
- 与企业客服范围无关、且不需要模型或知识检索的问题，应在公开消息链路中得到稳定范围拒绝，并且不产生受信任来源审计。

## 非目标

- 不实现复杂国际电话号码库；本地验收先支持中国大陆 11 位手机号 `1[3-9]\d{9}`，其它格式视为非法。
- 不把所有闲聊都拒绝；只对确定无关且风险较高的问题做稳定拒绝，例如彩票预测、其他公司未公开商业机密。
- 不新增 OpenAPI 字段；错误仍走现有 `ErrInvalidArgument` 映射。

## 设计

手机号校验放在 `AICCPublicService.submitLeadValues`。服务端按已配置字段的 `field_type` 校验提交值，`phone` 类型要求匹配 `1[3-9]\d{9}`，`email` 类型继续使用轻量格式校验。前端公开留资表单在提交前执行同样规则，给访客即时错误提示；后端仍是最终保护层。

暂停客服启动逻辑放在 `AICCService.SetAgentStatus`。当 action 为 `start` 时，读取企业 AICC 配置 revision；如果当前 agent 的 `applied_config_revision` 落后，则创建针对该 agent 的重启/初始化任务，并在任务成功后由既有 worker 写入 `applied_config_revision`。如果当前实现已有可复用的 AICC runtime restart job，则复用它，避免新增任务类型。

无关问题拒绝放在公开消息服务进入模型调度前。对极少量确定性高的无关风险输入做规则判断，例如彩票预测、未公开商业机密、赌博预测等，直接返回客服范围拒绝文本，保存正常访客/助手消息，但不调用 Hermes、不写来源审计。这样 E2E 可以稳定验证边界，而不依赖模型自由输出。

## 验证

- Go service 单测覆盖手机号合法/非法、无关问题拒绝不调用 chat、暂停启动落后 revision 会触发应用。
- Vue 单测覆盖公开留资表单非法手机号不调用 API。
- Playwright focused E2E 覆盖非法手机号不落正式线索、暂停启动后 revision 收敛、无关问题拒绝且来源审计为零。
