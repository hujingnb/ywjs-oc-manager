# 定制技能(需求工单 → 人工定制 → 定向交付)— 设计文档

## 背景与目标

现有 skill 体系已下沉到**实例(app)级**:运行时 skill = `app_skills`;可浏览安装的
**市场**由可扩展**来源(SkillSource)**聚合,首批 `platform`(平台库,管理员上传,
**全局可见**)与 `clawhub`(公共库)。详见 `docs/superpowers/specs/2026-06-03-skill市场-design.md`。

但平台库是**全局**的——所有人都能在市场看到。本设计要解决的是**定向定制**需求:

1. 企业用户能**提交对 skill 的需求**(完整工单:状态/多轮对话/附件/报价)。
2. 平台管理员后台**人工接单、定制、交付**。
3. 交付出来的定制技能**按提交者角色定向可见**:
   - 企业**成员(org_member)**提的 → 推给该企业**全部用户**(含管理员)。
   - 企业**管理员(org_admin)**提的 → 仅推给该企业**管理员**。
4. 未来是**收费服务**(首版仅留报价字段,不接支付)。

设计原则:**扩展现有 skill 市场,不另起炉灶**。定制技能作为市场新来源 `custom` 接入
现有 `SkillSource` 抽象,安装/更新完全复用 `app_skills` 现有链路;只新增「需求工单」
模块与「定向可见性」过滤。

## 术语

- **工单(skill ticket)**:一条 skill 定制需求及其全过程(对话、附件、状态、报价)。
- **定制技能(custom skill)**:工单交付产出的 skill 归档,挂在新来源 `custom` 下,
  按目标范围对一群用户可见。结构镜像 `platform_skills`(多版本共存)。
- **目标范围(target / audience)**:定制技能对谁可见。`all_org`(整企业)/
  `org_admins`(仅企业管理员)/ `requester_only`(仅提交者)。默认据提交者角色,
  平台管理员交付时可调(可多 org)。
- **提交者角色快照(requester_role)**:提交工单**当时**的角色,用于决定默认目标范围;
  即使日后角色变化,默认范围以提交时为准。

## 已确认决策(brainstorming 2026-06-10)

1. **交付语义**:不自动装。交付产出 `custom` 来源 skill,出现在「技能市场›定制」筛选;
   用户一键安装进自己实例(`app_skills`),重启生效。复用现有安装链路。
2. **可见范围**:默认按提交者角色(member→整企业、admin→仅管理员);平台管理员交付时
   可调(收窄「仅本人」、改「仅管理员」、加组织)。锁在提交者所在 org(可扩多 org)。
3. **工单形态**:完整工单 — 状态机 + 多轮评论 + 附件 + 报价 + 打回/重开。
4. **收费**:首版仅工单留「报价金额」字段(管理员填、用户可见),**不接支付/扣费**。
5. **命名**:功能叫**「定制技能」**(用户侧 tab + 管理员后台菜单);市场来源筛选标签
   叫**「定制」**(与「平台库 / ClawHub」并列)。
6. **用户侧入口**:「技能」页第三个 tab「定制技能」(已安装 / 技能市场 / 定制技能)。
7. **用户侧布局**:工单列表(标题/状态/报价/「去安装」)+ 提交表单(标题/描述场景/
   可选附件)+ 工单详情 = **右侧抽屉**。
8. **「去安装」**:跳「技能市场」tab,自动选中「定制」筛选并定位到该卡片。
9. **工单 vs skill 可见性**:工单(申请+对话)**提交者私有**(仅他 + 平台管理员可见);
   交付的 **skill** 才按目标范围对一群人可见。
10. **发现性**:**不做任何端用户提醒**(用户自己逛市场「定制」筛选发现)。
11. **卡片展示**:**显示申请人**「由 X 申请」+ **范围徽章**(整企业可见 / 仅管理员可见)。
12. **管理员后台**:平台库旁新菜单「定制技能」→ 工单队列(带提交者 org/角色,按状态/
    组织筛选)→ 详情**抽屉(宽)**→ 操作(改状态/报价/拒绝/回复)→ 交付弹窗。
13. **交付上传**:复用现「上传平台技能」交互——**粘贴 Markdown(单个 SKILL.md 全文)或
    上传文件夹**(webkitdirectory),前端解析预览 `name`/`description`、填描述
    (默认取自 SKILL.md 可改),**前端校验并打包成扁平 tar** 后提交。`name` 取自 SKILL.md
    的 `name` 字段(非单独输入)。**不是直接上传 tar。** **与平台库不同:版本号不手填**
    (见决策 17 自动生成)。
14. **版本迭代**:定制技能多版本共存(`UNIQUE(name, version)`,同平台库);一工单 →
    一定制技能(按 `name`)→ 多次交付 = 多版本,范围沿用,旧版本内容仍缓存对象存储;
    沿用市场 `latest_version > version`「更新」机制,用户在「已安装」**自行点更新**,
    不自动推。
    - **技能名一致性校验(后端强制)**:工单首次交付时,把解析出的 `name` 写入
      `skill_tickets.custom_skill_name` 锁定。**再次交付(迭代)时,后端必须校验本次解析出的
      `name == custom_skill_name`,不一致直接拒绝交付**并提示「迭代必须沿用同一技能名
      `<custom_skill_name>`」。前端在交付弹窗也做同样校验提前拦截,但**以后端为准**
      (前端可绕过)。这保证迭代是「同名加版本」,不会误建出第二个定制技能。
15. **工单生命周期(事件驱动)**:交付即**自动关闭**(delivered);用户提交问题/反馈则
    **自动重新打开**回处理队列(pending);管理员再交付 → 再自动关闭。循环,无需手动关。
    - **拒绝后也能重开**:`rejected` 工单**对提交者保持可继续操作**——用户补充评论/上传
      附件即自动回 `pending`(角标 +1),交回平台管理员重新处理。拒绝不是终态死路。
16. **通知处理人员**:**仅后台「定制技能」菜单待处理工单数角标**(新工单 + 用户反馈重开
    均计入)。项目无任何现成邮件/webhook/通知设施,故不做外部推送。
17. **版本号自动生成**:定制技能版本号**不手填**,交付时**后端按上传时间自动生成**(格式
    `YYYYMMDD-HHmmss`,UTC,保证唯一与可读)。可行性依据:平台库/定制技能的「最新版本」由
    `created_at DESC` 决定(`skill_update_checker.go` 同名首条即最新),**不解析版本串大小**,
    故时间戳版本完全兼容现有「更新」机制;`CustomSource` 的 latest 判定镜像平台库
    `buildPlatformLatestMap`(按 name、created_at DESC 取首条)。`UNIQUE(name, version)`
    靠秒级时间戳避免碰撞(人工交付节奏下不会同秒;极端碰撞时自动加后缀重试)。

## 整体架构

```
  用户(member/admin)                平台管理员                    目标用户群
        │                               │                            ▲
        │① 提交需求工单                  │                            │⑤ 市场「定制」筛选
        │  (标题/描述/附件)              │② 后台接单/报价/对话          │  一键安装→app_skills
        ▼                               ▼                            │
   ┌──────────────┐ ③ 交付(粘贴MD/传文件夹→前端打包 + 确认范围) ┌──────────────────┐
   │ skill_tickets │ ─────────────────────────────▶ │ custom_skills     │
   │ + comments    │   ④ 写库+对象存储, ticket=delivered │ + custom_skill_  │
   │ + attachments │ ◀───── 用户反馈→ticket=pending ──── │   targets         │
   └──────────────┘                                  └────────┬─────────┘
                                                              │ CustomSource(按 org+受众过滤)
                          市场来源抽象 SkillSource ◀───────────┘
                          platform / clawhub / 【custom 新增】
```

定制技能复用市场来源抽象作为第三个来源;工单模块是其上游「生产线」。

## 数据模型

> 建表规范:`ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci`,
> **每个字段与表都带 SQL `COMMENT`**。迁移号 `000010`。时间用 `DATETIME(6)`,
> 回填注意 app 连接固定 UTC(见既有时区约定,迁移内用 `UTC_TIMESTAMP(6)` 而非 `NOW()`)。

### skill_tickets(工单主表)

```sql
CREATE TABLE skill_tickets (
    id                CHAR(36)     NOT NULL                 COMMENT '主键 UUID',
    org_id            CHAR(36)     NOT NULL                 COMMENT '提交者所属组织,定向范围锚点',
    requester_user_id CHAR(36)     NOT NULL                 COMMENT '提交者 user id',
    requester_role    VARCHAR(32)  NOT NULL                 COMMENT '提交时角色快照:org_admin|org_member,决定默认目标范围',
    title             VARCHAR(255) NOT NULL                 COMMENT '需求标题',
    description        TEXT         NOT NULL                 COMMENT '需求描述/使用场景/输入输出示例',
    status            VARCHAR(16)  NOT NULL DEFAULT 'pending' COMMENT '状态:pending|processing|delivered|rejected',
    quote_amount_cents BIGINT      NULL                    COMMENT '管理员报价金额(分),NULL=未报价;用分存避免小数,展示层 /100 格式化为元。首版仅展示不扣费',
    custom_skill_name VARCHAR(128) NULL                     COMMENT '首次交付时写入,关联 custom_skills.name,一工单一技能',
    reject_reason     VARCHAR(512) NULL                     COMMENT '拒绝原因(status=rejected 时)',
    created_at        DATETIME(6)  NOT NULL DEFAULT CURRENT_TIMESTAMP(6) COMMENT '提交时间',
    updated_at        DATETIME(6)  NOT NULL DEFAULT CURRENT_TIMESTAMP(6) ON UPDATE CURRENT_TIMESTAMP(6) COMMENT '最后更新时间(状态变更/评论/交付)',
    PRIMARY KEY (id),
    CONSTRAINT fk_skill_tickets_org FOREIGN KEY (org_id) REFERENCES organizations(id),
    CONSTRAINT fk_skill_tickets_requester FOREIGN KEY (requester_user_id) REFERENCES users(id),
    KEY idx_skill_tickets_status_updated (status, updated_at DESC),
    KEY idx_skill_tickets_requester (requester_user_id, created_at DESC),
    KEY idx_skill_tickets_org (org_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci
  COMMENT='skill 定制需求工单,人工处理生命周期';
```

### app_skills.source 扩展

`app_skills.source` 既有取值 `platform | clawhub`,**追加 `custom`**。安装定制技能时
`source='custom'`、`source_ref=custom_skills.name`,其余字段(缓存 tar、sha256、版本、
快照)走现有安装链路,运行时与其他来源无差别。对象存储 key 复用 `library/` 前缀的
`LibrarySkillKey("custom", name, version, "tar")`。

## 市场来源:CustomSource

实现现有 `SkillSource` 接口,作为第三个来源:

```go
// CustomSource 把定制技能(custom_skills + custom_skill_targets)接入市场来源抽象。
// 与 platform/clawhub 不同:List/Get 必须按调用主体的 org_id + 角色过滤目标范围,
// 只返回对该主体可见的定制技能。
type CustomSource struct { /* custom_skills store + targets store */ }

func (s *CustomSource) Kind() string { return "custom" }
// List 仅返回 principal 可见(命中 custom_skill_targets)的最新版本条目,
// 每条带 requester 展示名(「由 X 申请」)与范围徽章信息。
// Get/LatestVersion/FetchArchive 与 platform 同构(按 name+version 取归档)。
```

**可见性过滤(核心)**:对 `Principal{OrgID, Role, UserID}`,命中条件:
存在 `custom_skill_targets` 行满足 `org_id = p.OrgID` 且
- `audience = 'all_org'`,或
- `audience = 'org_admins'` 且 `p.Role = org_admin`,或
- `audience = 'requester_only'` 且 `p.UserID = ticket.requester_user_id`。

平台管理员不通过此来源浏览(其入口在后台工单);该过滤只服务企业用户的市场视图。

**版本/最新判定**:定制技能版本号交付时自动按上传时间生成(决策 17)。「最新版本」镜像
平台库:同名按 `created_at DESC` 取首条,**不解析版本串**。`LatestVersion(ref)` 即返回
该 name 下 `created_at` 最新行的 version。**`skill_update_checker.go` 需新增 `custom`
分支**:在 `resolveLatest` 加 `case "custom"`,构建 name→最新版本 map(同
`buildPlatformLatestMap` 逻辑,数据源换成 `custom_skills`),其余回写流程复用。

## 工单状态机

```
           提交                 接单/报价              交付
  (无) ──────────▶ pending ──────────▶ processing ──────────▶ delivered(关闭)
                     │                    │                      │
                     │  拒绝              │  拒绝                │  用户提交反馈/问题
                     ▼                    ▼                      ▼
                 rejected ◀───────────────┘                  pending(重开)
```

- **pending**:待处理。新工单初始态;用户反馈重开后回到此态。计入后台角标。
- **processing**:制作中。管理员接单/报价后。**不计角标**(已被人接手)。
- **delivered**:已交付(关闭)。至少交付过一个版本。**不计角标**。
- **rejected**:已拒绝(带 `reject_reason`)。**不计角标**。可被用户重开为 pending。
- **重开**:`delivered` 或 `rejected` 工单,提交者新增评论(反馈/追加需求)→ 自动回
  `pending`,角标 +1。
- **后台待处理角标** = `COUNT(status = 'pending')`。新工单与用户反馈重开都落在 `pending`,
  故二者均计入;管理员接单转 `processing` 后从角标移除,避免长期占角标。

## 权限(`internal/auth/authorizer.go`)

沿用集中式 `Can*` 谓词,基于 `domain.UserRole{PlatformAdmin,OrgAdmin,OrgMember}`:

```go
// CanSubmitSkillTicket 判断能否提交定制需求工单:企业成员或企业管理员(平台管理员不提需求)。
func CanSubmitSkillTicket(p Principal) bool {
    return p.Role == domain.UserRoleOrgAdmin || p.Role == domain.UserRoleOrgMember
}

// CanManageSkillTicket 判断能否处理/交付/拒绝工单:仅平台管理员。
func CanManageSkillTicket(p Principal) bool {
    return p.Role == domain.UserRolePlatformAdmin
}

// CanViewSkillTicket 判断能否查看某工单详情/对话:提交者本人 或 平台管理员。
// 工单对企业内其他人不可见(可见的只有交付出来的 skill)。
func CanViewSkillTicket(p Principal, ticketRequesterUserID string) bool {
    return p.Role == domain.UserRolePlatformAdmin || p.UserID == ticketRequesterUserID
}
```

定制技能上传交付沿用平台库的 `CanManagePlatformSkill` 同级语义(平台管理员),或新增
`CanDeliverCustomSkill`(= 平台管理员)。市场可见性过滤在 `CustomSource`/service 层做。

## 前端界面(Vue3 + Naive UI + TanStack Query)

复用现有 skill 前端结构(`useSkills.ts`、`SkillManager.vue`、市场卡片)。

1. **用户侧「定制技能」tab**(技能页第三个 tab):
   - 工单列表表格:标题 / 状态徽章 / 报价 / 更新时间;已交付行带「去安装」→ 跳市场定制筛选。
   - 「+ 提交需求」→ 表单(标题 + 描述 + 可选附件多文件)。
   - 点工单 → **右侧抽屉**:状态/报价 + 对话气泡流 + 回复框 + 附件;已交付显「去安装」。
     **回复框/附件在任何状态都可用**(含 `delivered`、`rejected`):提交者发评论或传附件即触发
     重开 → 工单回 `pending`。`rejected` 抽屉额外展示拒绝原因 + 「补充说明重新提交」提示。
2. **技能市场「定制」筛选**(`SkillManager.vue` 市场视图加来源 chip):
   - 来源 chip:全部 / 平台库 / ClawHub / **定制**。
   - 定制卡片:技能名 + 定制徽章 + 描述 + 版本 + **范围徽章**(整企业可见/仅管理员可见)
     + **「由 X 申请」** + 安装/已装/更新按钮(复用现有安装与 `latest_version` 更新逻辑)。
3. **平台管理员后台「定制技能」**(平台库菜单旁,菜单项带**待处理角标**):
   - 工单队列:标题 / 提交者(姓名·组织·角色)/ 状态 / 报价 / 时间;按状态、组织筛选。
   - 点工单 → **宽抽屉**:左需求+附件+对话+回复框,右操作区(改状态、报价输入、拒绝)。
   - 「交付」→ 弹窗(复用上传平台技能交互:**粘贴 Markdown / 上传文件夹** → 前端解析预览
     `name`/`description`、填描述 → 前端打包扁平 tar 提交;`name` 取自 SKILL.md,**版本号
     不填,后端按上传时间自动生成**)
     + **目标范围**编辑(默认据提交者角色,可改 all_org/org_admins/requester_only、加组织)
     → 确认交付。**再次交付时解析出的 `name` 必须等于工单已记录的 `custom_skill_name`**
     (保证同名加版本而非新建技能)。

## 收费(首版范围)

- 工单 `quote_amount` 字段:平台管理员在详情/交付时填写,提交者在工单可见。
- **不做**:支付、余额扣减、订单、发票。数据模型已为未来留位(报价在工单上),接支付时
  再加订单/支付表,不改本设计已建表。

## 不做 / 明确排除

- **不自动安装/推送**:交付只让 skill 在市场可见,用户自行安装/更新。
- **不接支付**:仅报价字段。
- **不做外部通知**:仅后台 `pending` 工单角标;无邮件/webhook/站内信。
- **不做端用户发现提醒**:无红点、无通知。
- **不暴露工单给企业内非提交者**:工单私有,仅交付的 skill 定向可见。
