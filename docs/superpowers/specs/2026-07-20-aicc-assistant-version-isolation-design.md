# AICC 与助手版本业务隔离设计

## 背景

当前智能客服在创建时会自动建立一个隐藏 AICC 应用，并把企业
`assistant_version_ids` 中的第一个助手版本写入 `apps.version_id`。随后初始化与
bootstrap 从该版本读取主模型、提示词、智能路由和行业知识库关联。虽然 AICC 已使用
独立运行时镜像和固定安全 Skill，配置数据仍依赖普通助手业务。

智能客服与普通助手是两个独立业务。AICC 应由平台为企业配置统一模型，由企业自行配置
每个智能体的人设与知识范围，不应再读取或绑定助手版本。

## 目标

- AICC 与 `assistant_versions`、企业助手版本白名单完全解除运行时和业务依赖。
- 平台管理员在企业管理中开通 AICC 时配置企业统一使用的单一模型。
- 企业管理员创建或编辑智能体时配置客服人设、业务场景、欢迎语、回答边界和知识范围。
- 平台修改企业 AICC 模型后，正在运行的智能体逐个静默重启并立即切换到新模型。
- 普通实例的助手版本、智能路由、Skill 和行业知识库行为保持不变。
- 已有 AICC 企业从当前首个助手版本一次性迁移模型，保证现有服务连续性。

## 非目标

- 不建立独立的“客服版本”目录或客服版本授权体系。
- 不允许企业管理员或单个智能体选择不同模型。
- AICC 不支持智能路由。
- 不把普通助手版本的 Skill 下发给 AICC；客服继续使用专用镜像内置的固定安全白名单。
- 本次不改变 AICC 专用镜像、公开聊天协议、会话数据结构或普通实例运行时。

## 总体架构

系统形成两条互不交叉的配置链路：

1. 普通实例继续读取 `organizations.assistant_version_ids`、`assistant_versions` 和普通
   Hermes 运行时镜像。
2. 智能客服只读取企业 AICC 配置、智能体配置、企业行业知识库授权和 AICC 专用镜像。

AICC 隐藏 `app` 只承担运行时身份、密钥、状态和 Kubernetes 资源锚点职责。其
`version_id` 必须为空，AICC 初始化和 bootstrap 不得调用 `GetAssistantVersion`。

## 数据模型

### 企业 AICC 配置

新增 `organization_aicc_configs`：

| 字段 | 约束与用途 |
|---|---|
| `org_id` | 主键并外键关联 `organizations.id`，每个企业最多一份配置 |
| `enabled` | 是否开通 AICC |
| `model` | 企业全部智能客服统一使用的模型；启用时必须非空 |
| `agent_limit` | 智能体数量上限，`NULL` 表示不限，不能小于零 |
| `revision` | 配置修订号；影响运行时的字段变化时递增 |
| `created_at` | 创建时间 |
| `updated_at` | 最近更新时间 |

启用状态下数据库约束要求 `model` 非空。禁用时保留最近一次模型，便于重新开通，但任何
运行时入口仍必须检查 `enabled`。

迁移完成后删除 `organizations.aicc_enabled` 和 `organizations.aicc_agent_limit`，防止出现
两个配置源。对外组织视图可以继续聚合返回相同语义的 AICC 字段，但写操作必须进入独立
AICC 配置服务。

### 智能体配置

`aicc_agents` 增加：

| 字段 | 约束与用途 |
|---|---|
| `persona` | 企业管理员维护的客服人设文本；允许为空以兼容历史智能体 |
| `applied_config_revision` | 当前运行时已应用的企业 AICC 配置修订号 |

行业知识库选择继续存放在 `aicc_agent_knowledge`。平台先通过
`organization_industry_knowledge_bases` 授权给企业，企业管理员只能为智能体选择授权范围
内的行业库。

## 权限与 API

新增独立企业 AICC 配置接口：

- `GET /api/v1/organizations/:id/aicc-config`
- `PUT /api/v1/organizations/:id/aicc-config`

平台管理员可以读取和修改任意企业配置。企业管理员只能读取本企业配置，不能修改模型、
开通状态或数量上限。权限判断继续集中在 `internal/auth/authorizer.go`。

更新接口接收 `enabled`、`model` 和 `agent_limit`。启用 AICC 或修改模型时，后端必须通过
现有模型目录校验模型当前可用。模型变更和 rollout 任务入库必须处于同一事务。

智能体创建和更新请求增加：

- `persona`
- `industry_knowledge_base_ids`

响应回显上述字段。请求中不增加 `version_id`、模型或路由字段。后端必须验证所有行业库
均已授权给目标企业，并在同一业务事务中整组替换智能体的行业库选择。

## 创建与启动流程

### 创建智能体

1. 校验操作者具有目标企业 AICC 管理权限。
2. 读取 `organization_aicc_configs`，要求配置存在、已启用且模型有效。
3. 校验智能体数量限制和行业知识库授权。
4. 创建 `app_type=aicc` 的隐藏应用，`version_id=NULL`。
5. 创建 `aicc_agents` 记录和知识范围，并入队 `app_initialize`。
6. 任一步骤失败时沿用现有补偿机制软删除已创建的隐藏应用，避免孤儿运行时。

### AICC 初始化

`app_initialize` 必须在普通应用和 AICC 分支进入版本加载之前完成分流：

- 普通应用仍要求 `version_id` 并加载助手版本。
- AICC 根据 `app_id` 查询智能体及企业 AICC 配置，不读取 `version_id`。
- AICC 镜像继续由 `aicc.runtime_image` 解析。
- AICC 不执行助手版本 Skill 种子注入。

初始化完成时，把目标企业配置修订号写入智能体的 `applied_config_revision`。修订号只在 pod
Ready 且 bootstrap 已成功应用配置后落库，不能提前表示成功。

### Bootstrap 与提示词

AICC bootstrap 按隐藏 `app_id` 查询智能体和企业配置，使用：

- `organization_aicc_configs.model` 作为唯一模型；
- 固定为空的 routing；
- AICC 专用镜像内置的安全 Skill；
- 智能体选择的企业与行业知识范围。

提示词按以下顺序渲染：

1. 平台固定 AICC 安全规则，优先级最高且企业不可覆盖；
2. 智能体 `persona`；
3. 智能体 `scenario`；
4. 智能体 `answer_boundary`。

`greeting` 仅作为访客进入页面时的欢迎消息，不拼入系统提示词。

## 知识检索

AICC 检索不得再通过 `apps.version_id` 调用
`ListIndustryKnowledgeBasesByAssistantVersion`。检索范围以 `aicc_agent_knowledge` 为唯一业务
配置来源，并继续执行租户和授权校验：

- 当前智能体知识；
- 当前企业知识；
- 当前智能体明确选择且企业仍获授权的行业知识库。

平台撤销企业行业库授权时，沿用现有清理逻辑立即删除相关智能体关联。普通实例仍按助手
版本关联检索行业知识库，不受本次调整影响。

## 模型变更与静默重启

修改企业模型时：

1. 在事务中更新模型并递增 `revision`。
2. 同一事务创建企业级 `aicc_model_rollout` 后台任务，payload 包含 `org_id` 和目标修订号。
3. 接口在配置与任务持久化成功后返回，不同步等待全部 pod 重启。
4. rollout 只选择未删除、正在运行且 `applied_config_revision` 落后于目标修订的智能体。
5. 每次只重启一个智能体，等待其 pod Ready 并应用目标修订后再处理下一台。
6. 暂停的智能体不被自动启动，下次启动时直接应用最新配置。
7. 多次连续修改时，执行器读取企业最新修订并合并过期任务，避免为中间模型重复重启。

单个智能体重启失败时任务保留并按现有后台任务退避策略重试，不继续重启下一台。任务错误
必须包含企业、智能体和失败阶段，供现有后台任务页面排查。已应用目标修订的智能体在任务
重试时必须跳过。

## 管理界面

### 平台企业管理

- AICC 配置作为企业详情中的独立区域展示。
- 开启 AICC 后模型必选，同时配置智能体数量上限。
- 模型候选项来自平台当前可用模型列表。
- 修改模型时显示确认弹框，明确保存后将逐个静默重启正在运行的智能体。
- 企业管理员只能查看当前企业客服模型。

### 智能体创建与编辑

- 新增“客服人设”多行文本框。
- 保留名称、业务场景、欢迎语、回答边界、隐私和投放配置。
- 增加企业已授权行业知识库多选。
- 不展示助手版本、智能路由或模型选择。
- 只读展示企业统一客服模型，说明模型由平台统一配置。

模型 rollout 进度复用现有后台任务体系，不新增独立进度页面。

## 数据迁移

迁移按以下顺序执行：

1. 创建 `organization_aicc_configs`，为 `aicc_agents` 增加新字段；每个现有企业均创建配置
   行，原样保留旧 `enabled` 和 `agent_limit` 值。
2. 对存在有效首个助手版本的企业，从该版本复制 `main_model`；已开通 AICC 的企业必须复制
   成功，未开通企业允许模型为空。该复制只发生一次。
3. 已有智能体的 `persona` 初始化为空；原有场景、欢迎语和回答边界不变。
4. 把 AICC 隐藏应用原 `version_id` 写入仅供迁移回滚使用的
   `aicc_version_isolation_backups(app_id, version_id)`；该表不建立助手版本外键，也不被应用
   代码读取。
5. 企业配置全部回填成功后，将 AICC 隐藏应用的 `version_id` 清空。
6. 删除组织表旧 AICC 字段。

已开通企业不存在首个有效助手版本时迁移必须失败并报告企业 ID，不得回退到平台默认模型。
上线前先通过只读预检列出此类企业并修正数据。

为保证回滚可恢复原始绑定，迁移保留仅供回滚使用的版本映射；该映射不建立助手版本外键，
不参与任何运行时或业务查询，也不阻止助手版本删除。经过约定的发布稳定观察期后，再用独立
清理迁移删除该映射。回滚时先恢复组织旧字段及隐藏应用原 `version_id`，再删除新配置结构。

## 异常处理

- AICC 配置不存在、未启用、模型为空或模型不可用时，拒绝创建和启动智能体。
- 选择未授权行业知识库时，拒绝创建或更新，不做部分保存。
- 模型配置更新与 rollout 任务任一写入失败时回滚整个事务。
- rollout 失败不把未处理智能体批量重启，也不重复处理已成功智能体。
- AICC bootstrap 不得在配置缺失时回退助手版本或普通默认模型。
- 普通应用仍在版本缺失时返回原有错误，AICC 分支不能改变其错误语义。

## 测试与验收

### 后端单元测试

- 已开通企业必须配置有效 AICC 模型。
- 创建智能体不读取企业助手版本白名单或助手版本记录。
- AICC 隐藏应用 `version_id` 为空，普通实例版本逻辑不变。
- bootstrap 使用企业模型和智能体人设，并保持平台 AICC 安全规则优先。
- AICC 不注入助手版本路由、Skill 或行业知识库。
- 行业知识库只能选择企业已获授权的条目。
- 修改模型递增修订并原子创建 rollout 任务。
- rollout 严格逐台执行，Ready 后再处理下一台，暂停智能体不被启动。
- rollout 重试跳过已应用目标修订的智能体，并合并过期修订。

所有新增测试和子测试按仓库规范添加相邻中文业务场景注释，并使用 `testify` 断言。

### 迁移测试

- 从企业首个助手版本正确回填模型。
- AICC 隐藏应用清空版本绑定，普通实例绑定不变。
- 已开通但没有有效版本时明确失败。
- 回滚恢复组织旧字段和隐藏应用原版本绑定。

### 前端与浏览器验收

- 平台管理员开通 AICC 时必须选择模型。
- 修改模型时出现静默重启确认提示。
- 企业管理员只能查看模型。
- 创建和编辑智能体可以保存、回显客服人设与授权行业知识库。
- 创建页面不出现助手版本或路由信息。
- 使用真实浏览器创建智能体，通过公开客服页面验证回复遵循配置的人设。
- 修改企业模型后验证智能体自动恢复服务并使用新模型继续对话。
- 定向验证普通实例创建和助手版本切换，确认两个业务互不影响。

浏览器验收默认使用定向、无头 Playwright spec，不执行无关全量 E2E。

## 完成标准

- AICC 生产代码中不存在通过 `apps.version_id`、`assistant_version_ids` 或
  `assistant_versions` 获取模型、提示词、路由、Skill 或行业知识库的路径。
- 平台可以为企业独立配置单一 AICC 模型，企业可以维护每个智能体的人设和授权知识范围。
- 模型变更能通过持久化任务逐台静默重启运行中的智能体，并可安全重试。
- 已有 AICC 数据完成无默认兜底的一次性迁移。
- 普通助手版本业务行为和相关定向测试保持通过。
