# 成员密码重置弹窗设计

## 背景

成员页当前先通过浏览器原生 `window.prompt` 收集新密码，再打开通用的
`ConfirmActionModal`，要求管理员输入成员登录名完成二次确认。国际化迁移后，确认弹窗的
提示被错误替换为“输入新密码”，但按钮仍按成员登录名校验，导致管理员按界面提示输入合法
密码时无法点击“确认重置”。此外，原生提示框与二次弹窗连续出现，交互割裂，密码也缺少统一
的可见性控制。

## 目标

- 点击“重置密码”后直接打开站内专用弹窗，不再使用 `window.prompt`。
- 在弹窗内通过一个密码输入框收集新密码，默认隐藏内容，并允许管理员切换显示状态。
- 新密码少于 8 位或请求正在提交时禁用“确认重置”。
- 继续复用现有密码重置 API、mutation 与页面反馈，不修改后端契约。

## 非目标

- 不调整后端密码复杂度规则。
- 不增加二次密码确认输入框。
- 不改变成员启停、删除或其他 `ConfirmActionModal` 调用方。
- 不改动生产数据或在线上页面执行密码重置。

## 方案

新增专用的 `ResetMemberPasswordModal` 组件，由它负责密码输入和弹窗内校验。成员页只负责
维护目标成员、调用现有 mutation，以及处理成功或失败反馈。

不扩展 `ConfirmActionModal`：该组件的职责是让用户输入业务对象名称确认高风险操作，而密码
重置需要处理敏感输入、长度校验和输入值回传，两者的交互语义不同。也不把完整弹窗直接写进
`MembersPage.vue`，避免继续增加页面组件的表单职责。

## 组件与数据流

1. 管理员点击某一成员的“重置密码”，成员页将该成员保存为当前重置目标。
2. `ResetMemberPasswordModal` 根据目标成员是否存在决定是否显示，并展示成员登录名和原密码
   立即失效的风险说明。
3. 管理员在密码输入框中输入一次新密码。输入框使用密码类型，默认隐藏内容，并提供显示/隐藏
   切换。
4. 密码长度达到 8 位且未在提交时，确认按钮可点击；点击后组件通过 `confirm` 事件回传新密码。
5. 成员页将成员 ID 与新密码交给现有 `useResetMemberPassword` mutation。
6. 成功时关闭弹窗、清空组件密码状态并展示成功反馈；失败时保持弹窗打开和当前输入，展示现有
   错误反馈，允许管理员修改后重试。
7. 取消、遮罩关闭或成功关闭时均清空密码，避免敏感值在下一次打开时复用。

## 错误与边界处理

- 空密码和少于 8 位的密码不允许提交。
- 提交期间禁用确认按钮并展示 loading，防止重复请求。
- 成员目标为空时不渲染有效提交入口；成员页提交处理仍保留空目标兜底。
- API 失败不清空输入，方便管理员修正或重试；关闭弹窗后必须清空。
- 密码只保存在前端短生命周期组件状态中，不写入日志、URL、持久化存储或文档。

## 测试与验证

- 为专用弹窗增加组件测试，覆盖默认隐藏、显示切换、短密码禁用、合法密码解锁、提交回传、
  busy 禁用以及关闭后清空。
- 更新成员页测试，覆盖点击“重置密码”直接打开专用弹窗、不调用 `window.prompt`、提交正确的
  成员 ID 和密码、成功关闭及失败保留弹窗。
- 运行上述最小相关 Vitest 测试和前端类型检查。
- 使用真实 Chromium 的无头模式，对本地 manager 后台执行成员密码重置弹窗的定向验证；只验证
  弹窗交互与按钮状态，不向线上环境提交任何写操作。
