# 实例渠道 Tab 全渠道列表与 Logo 设计

## 背景

实例详情页的渠道 tab 当前只围绕默认 `wechat` 绑定流程展示。用户进入页面时看不到未来会支持哪些渠道，也无法区分“当前只支持微信”和“其他渠道不存在”。本次目标是在不改动绑定业务流程的前提下，把渠道能力边界直接展示在页面上。

## 目标

- 在实例详情页渠道 tab 中列出微信、企业微信、飞书、钉钉四个渠道。
- 每个渠道展示对应 logo、渠道名称和支持状态。
- 当前仅微信可操作，企业微信、飞书、钉钉置灰并标记“暂不支持”。
- 保留现有微信登录、二维码展示、刷新二维码、解绑和状态文案逻辑。

## 非目标

- 不新增企业微信、飞书、钉钉的后端 adapter、API、状态机或绑定流程。
- 不修改 OpenAPI 契约和前端生成类型。
- 不引入远程图片依赖，也不做真实品牌资源下载。

## 交互与视觉设计

采用“左侧渠道列表 + 右侧当前渠道详情”的 A 布局。

左侧列表固定展示四个渠道：

- 微信：正常色彩展示，默认选中，状态为“已支持”。
- 企业微信：灰色 logo、灰色文字，状态为“暂不支持”。
- 飞书：灰色 logo、灰色文字，状态为“暂不支持”。
- 钉钉：灰色 logo、灰色文字，状态为“暂不支持”。

右侧详情只展示微信绑定内容，复用当前页面的状态行、错误提示、二维码渲染、过期提示和操作按钮。用户点击暂不支持渠道时不切换详情、不发起请求；未支持渠道仅作为能力预告和边界说明。

logo 采用前端本地组件化的简化标识：微信使用绿色品牌感标识，其余未支持渠道以同形状灰化标识展示。这样避免远程资源失败影响页面，也方便单元测试稳定断言。

## 架构设计

前端新增一个小的渠道展示模型，集中描述渠道列表：

- `type`：渠道类型，例如 `wechat`。
- `name`：中文渠道名。
- `supported`：是否已支持。
- `statusLabel`：已支持或暂不支持。
- `logo`：本地渲染用的标识配置。

`AppChannelsTab.vue` 继续负责微信绑定流程编排。现有 `channelType` 默认值保持 `wechat`，当前登录、解绑和进度查询仍只读取微信渠道。渠道列表只影响页面展示和可用状态，不参与 API 参数扩展。

如果后续新增真实渠道，只需要把对应配置改为 `supported: true`，再接入实际 adapter 和后端 API 能力；本次不预留复杂动态注册机制。

## 数据流

页面加载后：

1. 左侧根据本地渠道配置同步渲染四个渠道。
2. 右侧按现有逻辑调用 `useChannelProgressQuery(appId, 'wechat')` 查询微信进度。
3. 用户点击“发起登录”或“刷新二维码”时，仍调用微信渠道的 `useBeginChannelAuth`。
4. 用户点击“解绑”时，仍调用微信渠道的 `useUnbindChannel`。
5. 未支持渠道不触发查询、登录或解绑 mutation。

## 错误处理

微信渠道的错误展示沿用现有 `progress.error_message` 和二维码过期提示。未支持渠道没有后端请求，因此不产生接口错误；页面通过“暂不支持”和置灰状态表达不可用原因。

## 测试计划

- 更新 `AppChannelsTab` 单元测试，断言微信、企业微信、飞书、钉钉全部渲染。
- 断言微信显示“已支持”并保持可操作入口。
- 断言企业微信、飞书、钉钉显示“暂不支持”且视觉上带禁用标记。
- 断言已绑定微信时，右侧仍展示中文状态和已绑定身份。
- 运行受影响的前端 Vitest 用例。
