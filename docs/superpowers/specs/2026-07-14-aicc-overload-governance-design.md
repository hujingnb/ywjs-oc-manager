# AICC 客服高并发过载治理设计

- 日期：2026-07-14
- 状态：已确认，待编写实施计划
- 范围：公开客服请求在 Hermes 运行时或模型上游高并发过载时的可靠接收、削峰、扩缩容、熔断、恢复与可观测性。

## 1. 目标与非目标

### 目标

1. 访客消息一经服务端确认接收，即使 Hermes Pod 缩容、重建或模型上游暂时拒绝，也不丢失、不重复产生完成回答。
2. 对 Hermes 运行时资源压力自动扩容，并在流量降低后自动缩容至至少一个热副本；不允许缩容到零副本。
3. 不因扩容放大对模型上游的冲击；上游过载时按受控并发、排队、退避、熔断和可选备用模型降级。
4. 同一会话的消息严格按顺序处理；任意健康 Pod 都能在原 Pod 已销毁时继续会话。
5. 访客和运营方均能看到可理解的当前状态，平台能据此告警和排障。

### 非目标

- 本期不改变 AICC 的业务问答、知识库、留资或人工客服能力。
- 本期不保证模型回答的语义完全一致；备用模型仅在后续具备配置时作为可选降级。
- 本期不实现 scale-to-zero 或按请求临时冷启动的运行模式。

## 2. 总体架构

公开客服请求不再直接把“客户端请求成功”与“模型调用已完成”绑定。AICC 接入 API 先将会话消息、幂等键和待执行任务持久化，再由独立调度器按容量投递给任一健康 Hermes Pod。

```text
网页挂件 / 公开链接
          │
          ▼
     AICC 接入 API
          │  事务写入
          ▼
会话与消息库 + 持久化任务队列
          │
          ▼
调度器（会话顺序、并发额度、重试与熔断）
          │
          ▼
  Hermes Pod × 1…N ─────► 模型上游
```

Pod 只负责执行，不是会话真相源。会话、消息、处理状态、幂等键和已完成回答均在外部持久化存储中保存。原 Pod 已缩容时，接入 API 或调度器读取最近完成的会话历史，投递给新的健康 Pod 继续处理。

## 3. 消息处理与状态

### 3.1 幂等与顺序

- 客户端为每条发送的访客消息生成 `message_id`；服务端以会话和该 ID 作为幂等约束。网络重试或重复点击只能返回已有消息状态，不得创建第二个模型任务。
- 接入 API 在同一数据库事务内写入访客消息、初始状态 `received` 以及队列任务。事务成功后才向访客确认“已接收”。
- 调度器为每个会话设置单消费者约束：仅当前一条未完成消息到达终态后，才能开始下一条，保证上下文不会乱序。

### 3.2 状态机

| 状态 | 含义 | 可达后继 |
|---|---|---|
| `received` | 访客消息已持久化，尚未取得调度资格 | `queued` |
| `queued` | 等待会话顺序、Hermes 容量和上游并发额度 | `processing`、`failed` |
| `processing` | 已由 Hermes 执行，可能正流式生成 | `completed`、`retry_wait`、`failed` |
| `retry_wait` | 可归因上游过载或缩容中断后的退避等待 | `queued`、`failed` |
| `completed` | 助手最终回答已持久化并可推送 | 终态 |
| `failed` | 不可恢复或超过重试上限 | 终态；访客可发起新的幂等重试 |

只在最终回答完整落库后进入 `completed`。已开始但未完成的流式结果不得被当作完成回答；需要恢复时回写为 `retry_wait`，由新执行者基于已完成上下文重试。

## 4. 容量与缩扩容

### 4.1 HPA

每个 AICC 客服运行时 Deployment 使用 HPA 自动调节副本数：

- `minReplicas=1`，永不缩至 0，确保随时可立即接待首条消息。
- CPU 平均利用率达到 70% 或内存利用率达到 75% 并持续约 1 分钟时扩容。
- 同时接入队列深度和 Hermes 在途请求数等业务指标，以便在 CPU 尚未升高的突发流量下提前扩容。
- 负载恢复后设置约 10 分钟缩容稳定窗口，避免副本数来回震荡。
- `maxReplicas` 必须由平台总资源和模型上游配额共同约束，不能仅根据 Kubernetes 资源无限放大。

### 4.2 优雅缩容

1. 被选中缩容的 Pod 先从服务发现中摘除，并停止从调度器领取新任务。
2. 已领取任务在 Kubernetes 优雅终止期限内继续完成，仍然正常持久化回答。
3. 到期未完成的任务必须带租约回收并回写 `retry_wait`，不能随 Pod 一同消失。
4. 新 Pod 或其他健康 Pod 重新领取任务，依据持久化会话历史继续执行。

## 5. 上游过载保护

### 5.1 判定

单次 `429`、明确的配额/限流错误、`503/529`、连接超时、首字节超时或总响应超时均标记为“可归因上游过载”。单次失败不立即宣告整个上游不可用；短时间连续失败才触发熔断。

默认熔断条件为：连续 5 次可归因失败，或 30 秒窗口内可归因失败率达到 30%。熔断持续 30 秒，然后以少量探测请求进入半开状态；探测成功则逐步恢复，失败则重新熔断。阈值需实现为平台可配置参数，并以该默认值作为初始设置。

CPU 和内存仅用于判断 Hermes Pod 是否拥塞，不能单独判断模型供应商是否过载；上游 HTTP 结果、延迟和队列积压才是上游过载的依据。

### 5.2 并发额度与重试

- 调度器在执行前同时获取上游、企业、机器人和会话四层并发额度。HPA 扩容不会绕过上游额度。
- 可归因过载使用指数退避并加入随机抖动，默认延迟为 2、5、10、20、40 秒，最多尝试 5 次。
- 仅未产出完成回答的任务可以自动重试。鉴权、请求格式、内容安全等确定性错误直接失败，不重试。
- 队列容量设为有界。达到上限时拒绝新的消息并提示访客稍后重试，避免无限排队耗尽数据库、队列资源或制造不可预期等待。
- 若后续配置备用模型，只有主上游已熔断且备用模型拥有独立额度时才可切换；否则继续排队或失败提示。

## 6. 用户反馈与可观测性

公开端根据消息状态显示“排队中”“正在重试”“回答已完成”或“当前繁忙，请稍后重试”。前端不得因轮询、断线重连或用户重复点击而盲目重新提交同一 `message_id`。

必须采集并告警以下指标：队列深度和等待时长、消息状态转移耗时、上游错误率和熔断状态、重试次数、各层并发额度使用率、Pod 在途请求数、HPA 扩缩容事件、缩容回收任务数、会话恢复次数及最终失败率。日志和指标需包含组织、机器人、会话、消息、上游和执行 Pod 的可关联标识，但不得泄露访客内容或敏感留资。

## 7. 验收与测试

1. 突发压测下，已确认接收的消息均能查询到，且同一 `message_id` 只有一条访客消息和至多一个完成回答。
2. 同会话连续发送多条消息时，模型调用顺序与发送顺序一致。
3. 模拟 `429`、`503/529` 和超时后，任务按退避重试；满足阈值时熔断，探测恢复后恢复投递。
4. 在任务执行中触发扩容和缩容，旧 Pod 摘流后不领新任务；未完成任务可由其他 Pod 恢复，已完成回答不重复生成。
5. 低负载后副本数会缩至 1，且任何时刻都不会为 0。
6. 队列满时平台有界拒绝新任务，既有任务不受影响，访客得到明确提示。
7. 业务、资源、上游失败和恢复指标均可在监控中观察，并能按机器人和上游定位。

## 8. 实施边界

实现时应保持 AICC 现有公开聊天、会话、消息及 Hermes 隐藏应用的契约兼容。持久化任务、状态、并发令牌和会话恢复须优先复用仓库现有数据库、Redis 异步队列和 Kubernetes 编排模式；实施计划阶段再根据现有数据模型确定最小新增表、接口和迁移范围。
