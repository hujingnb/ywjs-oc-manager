# 应用详情组织名称展示设计

## 背景

应用详情页的“所属组织”当前直接展示 `org_id`，标题旁也展示 `app.id`。这类 UUID 对业务用户没有明确含义，会降低页面可读性。项目中仍有一些技术标识用于运维诊断，例如容器 ID、Runtime Node ID、Job ID 和审计日志 target/actor ID，本次不扩大处理这些排障信息。

## 目标

- 应用详情页面向业务用户的主展示信息不直接暴露应用或组织 UUID。
- “所属组织”展示组织名称，而不是 `org_id`。
- 当组织名称无法加载时，展示友好兜底文案，不把 UUID 当作主要内容。
- 保持现有应用详情、权限判断、路由和运行时操作行为不变。

## 非目标

- 不调整后端 `AppResult` 契约，不新增 `org_name` 字段。
- 不重构审计日志、运行时、任务追踪等排障视图中的技术标识展示。
- 不修改 OpenAPI 生成产物。
- 不做全站 ID 展示治理。

## 方案选择

采用前端名称映射方案：应用详情页继续使用现有 `AppDTO.org_id` 做权限和查询上下文，同时通过现有组织查询能力取得组织名称并用于展示。

这个方案改动最小，不改变后端接口，也不需要同步 OpenAPI 和生成类型。平台管理员场景下可以复用组织列表数据；组织角色场景下可以按应用的 `org_id` 查询组织详情或利用现有组织查询 hook 补齐名称。

未采用后端补字段方案，原因是它需要修改 store 查询、service DTO、handler 测试、OpenAPI 和前端生成类型，超出本次“只处理明显业务展示问题”的范围。未采用只在模板里硬编码隐藏 UUID 的方案，因为它无法解决“所属组织”缺少可读名称的问题。

## 前端设计

应用详情父页面继续负责加载应用并向子 tab 注入 `app`。标题区域只显示应用名称和状态，不再在名称旁显示 `app.id`。

概览 tab 增加组织名称展示逻辑：

- 输入：父页面注入的 `app.org_id`。
- 查询：使用现有组织查询能力按 `org_id` 获取组织资料，或复用已有组织列表缓存中的名称。
- 展示：`所属组织` 显示组织名称。
- 兜底：组织名称加载中或不存在时显示 `未知组织`，避免将 UUID 暴露为主文案。

权限判断仍使用 `app.org_id` 和 `owner_user_id`，不受展示字段影响。知识库等依赖 `org_id` 的 API 调用也继续使用原 ID。

## 保留的技术标识

以下内容本次保留，因为当前用途偏排障或系统操作追踪：

- 容器 ID 和容器镜像。
- Runtime Node ID。
- Job ID 及 JobProgressPanel 的任务副标题。
- 审计日志里的 actor/target ID。
- 表格 row key、路由参数、API 请求参数中的 ID。

这些信息后续如需治理，应单独设计“业务名称 + 可复制技术 ID”的全站展示规则。

## 错误处理

- 组织查询失败时，应用概览仍正常展示应用其他字段。
- “所属组织”显示 `未知组织`，不阻断应用详情页操作。
- 如果应用本身加载失败，沿用现有应用详情错误提示。

## 测试与验证

- 补充或更新应用详情/概览相关前端测试，覆盖组织名称展示。
- 覆盖组织名称缺失时的兜底文案。
- 运行应用页相关测试，确认应用详情和应用列表行为不被破坏。
- 如仅前端展示变更，不运行 OpenAPI 生成命令。
