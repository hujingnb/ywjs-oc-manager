# Hermes v0.18.2 版本接入设计

## 背景

当前普通 Hermes runtime 变体使用上游日期 tag 作为目录名、配置 ID 和镜像版本，例如
`v2026.7.1`。上游 Hermes Agent 的产品版本则为 `v0.18.0`，导致用户可见版本与构建所需
Git ref 混用。

本次接入 Hermes Agent v0.18.2。其上游正式 Git tag 为 `v2026.7.7.2`。从本版本开始，
仓库内的 runtime 版本标识统一使用语义版本号，上游日期 tag 仅作为源码拉取 ref 保存。

## 目标

- 新增 `hermes-v0.18.2` runtime 变体，并将其设为普通实例的当前默认版本。
- 目录名、`version.txt`、配置 ID、用户可见标签和镜像 tag 统一使用 `v0.18.2`。
- 使用独立的 `hermes-ref.txt` 保存上游 Git ref `v2026.7.7.2`。
- 保留现有日期版本变体和配置项，供已有实例兼容及版本回退。

## 实现设计

新变体以当前 `hermes-v2026.7.1` 为基线复制，保留现有 oc-manager runtime 契约、补丁、
渲染器与测试。新目录中的 `version.txt` 写入 `v0.18.2`，`hermes-ref.txt` 写入
`v2026.7.7.2`。

Makefile 为普通 Hermes 变体新增上游 ref 派生逻辑：若当前变体存在 `hermes-ref.txt`，
则使用其内容作为 Docker 构建参数 `HERMES_REF`；否则回退到 `version.txt`。该回退保证
历史变体无需批量改名或补文件，仍可按原方式构建。版本校验继续以 `version.txt` 和目录名
为准，并额外校验显式上游 ref 非空。

默认构建变体切换为 `hermes-v0.18.2`。本地与生产 runtime image 列表新增 `v0.18.2`
并标记为当前版本，原 `v2026.7.1` 条目保留但移除当前标记。新镜像尚未发布时沿用项目现有
PENDING 占位约定，后续发布命令仍按语义版本镜像 tag 精确回填。

## 兼容性与边界

- 不重命名或删除历史变体，不改变已有助手版本引用的 image ID。
- AICC runtime 已独立使用 `hermes-ref.txt`，本次不升级其 Hermes 版本，也不改变其发布链路。
- 不修改 API 契约，因此无需重新生成 OpenAPI 与前端类型。
- 本次只登记并验证 runtime 变体，不执行生产镜像构建、推送或集群部署。

## 验证

- 先增加 Makefile 校验用例，证明语义版本与上游日期 ref 可独立解析，并验证历史变体回退。
- 运行新变体继承的 Python 单元测试，确认复制后的契约和工具逻辑保持通过。
- 运行 Makefile 版本守卫，确认目录名、语义版本和上游 ref 均合法。
- 检查配置解析及最终 git diff，确保没有混入无关文件。
