# manage 权限管理修复设计

## 背景

当前 manage 权限语义与产品预期不一致。平台管理员需要具备跨组织观察能力，但不应继承组织管理员或应用 owner 的写权限；组织管理员需要完整管理本组织成员、应用、用量、审计和组织知识库；组织成员只能查看和操作自己应用相关的信息。

现有代码已经把大部分权限谓词集中在 `internal/auth/authorizer.go`，但部分谓词把“可读应用”和“可管理应用/知识库/运行时操作”合并，导致平台管理员在一些路径上获得了不应拥有的写权限。前端菜单和按钮也沿用了同样的混合语义。

## 目标

- 平台管理员可以查看所有组织的成员、应用、用量和审计信息。
- 平台管理员不能自行创建应用，不能管理应用运行时、渠道绑定、工作区或自己的知识库，也不能写入组织/应用知识库。
- 组织管理员可以查看和管理当前组织下的成员、应用、用量、审计和组织知识库。
- 组织成员只能查看自己应用的用量、审计等信息，并只能访问自己拥有应用的相关资源。
- 权限判断继续集中在 `internal/auth/authorizer.go`，避免在 handler 或 service 内新增零散角色判断。
- 前端入口、按钮状态和后端 API 权限保持一致。

## 非目标

- 不引入数据库驱动的 RBAC 权限表。
- 不改变账号体系、组织模型、应用模型或 new-api 数据结构。
- 不重构无关页面布局、菜单设计或生成文件。
- 不手工修改 `openapi/openapi.yaml` 与 `web/src/api/generated.ts`，除非 handler 契约发生实际变化并通过生成命令更新。

## 角色权限矩阵

| 能力 | 平台管理员 | 组织管理员 | 组织成员 |
| --- | --- | --- | --- |
| 查看组织列表/组织详情 | 全部组织 | 仅本组织 | 仅本组织基础可见信息 |
| 管理组织/充值/运行节点 | 允许 | 不允许 | 不允许 |
| 查看成员列表 | 任意组织 | 本组织 | 不允许列表视角 |
| 管理成员 | 任意组织 | 本组织 | 不允许 |
| 查看应用列表/详情 | 任意组织 | 本组织全部应用 | 自己拥有的应用 |
| 创建应用 | 不允许 | 通过创建成员初始化本组织应用 | 不允许 |
| 应用运行时操作 | 不允许 | 本组织应用 | 自己拥有的应用 |
| 渠道绑定/工作区操作 | 不允许 | 本组织应用 | 自己拥有的应用 |
| 组织知识库读取 | 任意组织 | 本组织 | 本组织只读 |
| 组织知识库写入/同步重试 | 不允许 | 本组织 | 不允许 |
| 应用知识库读取 | 任意组织应用 | 本组织应用 | 自己拥有的应用 |
| 应用知识库写入 | 不允许 | 本组织应用 | 自己拥有的应用 |
| 用量查看 | 平台/任意组织/任意成员/任意应用 | 本组织/本组织成员/本组织应用 | 自己成员或应用维度 |
| 审计查看 | 任意组织/任意目标 | 本组织 | 自己作为操作者或自己应用相关的审计范围 |

## 后端设计

权限谓词集中在 `internal/auth/authorizer.go`：

- 保留 `CanViewOrg`、`CanManageOrg`、`CanViewMember`、`CanManageMember`、`CanEditMember` 的既有读写分离语义。
- 保留 `CanViewApp` 作为应用读取谓词，平台管理员返回 true。
- 调整 `CanManageApp`，平台管理员返回 false；组织管理员需同组织；组织成员需是 owner。
- 调整或新增组织知识库谓词：
  - `CanReadOrgKnowledge`：平台管理员、同组织管理员、同组织成员均可读。
  - `CanWriteOrgKnowledge`：仅同组织组织管理员可写。
  - `CanRetryOrgKnowledgeSync`：仅同组织组织管理员可触发。
- 调整应用知识库谓词：
  - `CanReadAppKnowledge`：沿用应用读权限。
  - `CanWriteAppKnowledge`：平台管理员不可写；组织管理员同组织可写；组织成员仅 owner 可写。
- 调整 `CanTriggerRuntimeOperation`：平台管理员不可触发；组织管理员同组织可触发；组织成员仅 owner 可触发。
- 如需要表达用量和审计范围，优先新增 `CanViewOrgUsage`、`CanViewMemberUsage`、`CanViewOrgAudit`、`CanViewOwnAudit` 等谓词，而不是在 service 内继续写裸 `principal.Role` 分支。

service 层只做业务流程和数据范围应用：

- `KnowledgeService` 使用组织知识库读写谓词，平台管理员只能读；同步状态重试按写权限限制。
- `UsageService` 将 app/member/org/platform 维度权限收束到 authorizer。普通成员不能看组织聚合用量，只能看自己的 member/app 维度；平台管理员可看所有读维度。
- `AuditService` 允许平台管理员和组织管理员按组织查询；组织成员不能使用组织审计列表，但必须支持自己的审计范围。实现上新增成员审计查询能力，按 `actor_id = principal.UserID` 或 `target_type = 'app' AND target_id = ownAppID` 过滤，只返回本人作为操作者或本人应用相关的记录。
- `RuntimeOperationService`、`ChannelService`、`WorkspaceService` 使用应用写/运行时谓词，避免平台管理员通过直接 API 写入应用状态或渠道。
- `OnboardingService` 和成员创建入口维持“平台管理员可创建组织成员”的能力，但前端不把它作为平台管理员自行创建应用的入口；若后端当前创建成员必然初始化应用，则平台管理员调用该复合接口应被禁止或改为只允许组织管理员调用，避免绕过“不能自行创建应用”。

## 前端设计

前端按后端权限语义隐藏或禁用操作，作为用户体验层防护：

- 平台管理员：
  - 菜单保留应用、用量、审计、组织、平台和运行节点等观察/平台管理入口。
  - 不显示“创建成员并初始化”这类会创建应用的入口。
  - 应用列表和详情可查看，但不显示重启、停止、删除、渠道写操作、工作区写操作、应用知识库上传/删除。
  - 组织知识库页面只读，不显示上传、删除、同步重试。
- 组织管理员：
  - 保留成员、应用、组织知识库、用量、审计管理入口。
  - 可操作本组织应用和知识库。
- 组织成员：
  - 应用列表只显示自己拥有的应用。
  - 用量页面只显示自己应用或成员维度。
  - 不显示成员管理、组织级审计、组织知识库写操作。
  - 审计入口展示“我的审计”视角，只加载自己作为操作者或自己应用相关的记录。

前端权限判断可以先使用小型 computed helper，后续如重复增多再抽取成 `web/src/domain/permissions.ts`，避免一次性引入过大抽象。

## 测试设计

后端单元测试：

- 更新 `internal/auth/authorizer_test.go`，覆盖三角色在应用读、应用管理、组织知识库读写、应用知识库读写、运行时操作上的矩阵。
- 更新 `KnowledgeService` 测试，覆盖平台管理员组织知识库只读、组织管理员可写、组织成员不可写。
- 更新 `UsageService` 测试，覆盖平台管理员跨组织读、组织管理员本组织读、组织成员只能读自己。
- 更新 `AuditService` 测试，覆盖组织成员不能读取组织审计列表，但能读取自己的审计范围。
- 更新运行时/渠道/工作区相关 service 测试，覆盖平台管理员 direct API 写操作被拒绝。
- 如调整 onboarding 后端权限，补充平台管理员不能通过复合创建接口初始化应用的测试。

前端验证：

- 跑类型检查和现有相关测试。
- 用浏览器分别登录：
  - `admin` / `admin123`
  - `test-org` / `test-org123`
  - `test-org-user1` / `test-org-user1`
- 验证菜单和按钮：
  - 平台管理员能查看跨组织应用/成员/用量/审计，但看不到应用写操作和知识库写操作。
  - 组织管理员能管理本组织成员、应用和知识库。
  - 组织成员只能看到自己的应用和自己的用量/审计范围。
- 对关键越权 API 做直接浏览器操作或网络请求验证，确认返回 403。

## 风险与取舍

- 平台管理员过去可能依赖应用写操作排障。新语义会禁止这类操作；如后续需要平台代运维能力，应单独设计“平台运维动作”并记录更强审计，而不是复用组织管理员权限。
- 普通成员审计范围需要新增查询能力。本次只支持本人 actor 记录和本人应用 target 记录，不扩展复杂筛选、跨目标组合查询或管理级过滤器。
- 禁止平台管理员通过 onboarding 创建应用可能改变现有平台管理员工作流，但这符合“不能自行创建应用”的边界。

## 验收标准

- `authorizer` 单测明确表达三角色权限矩阵并通过。
- 平台管理员直接调用应用写操作、组织知识库写操作、应用知识库写操作、运行时操作 API 返回 403。
- 平台管理员仍可查看所有组织的成员、应用、用量和审计。
- 组织管理员仍可管理本组织成员、应用和组织知识库。
- 组织成员无法看到组织管理入口，只能查看自己应用相关数据和自己的审计范围。
- 浏览器验证覆盖三个调试账号，发现问题后继续修复直到行为符合矩阵。
