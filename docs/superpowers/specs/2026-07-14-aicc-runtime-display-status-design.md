# AICC 运行时展示状态设计

## 目标

让智能客服工作台准确区分隐藏运行时 Pod 尚未可用，与 Pod 已就绪但客服尚未开始接待两种状态。用户不应把“草稿”误解为运行时启动过程。

## 状态模型

智能体接待意图与隐藏 App 的运行时状态保持两个独立维度：

- 智能体接待意图：现有 `draft`、`active`、`paused`、`deleted`，继续决定公开入口是否接受新会话。
- 运行时状态：关联隐藏 App 的 `status` 与 `runtime_phase`，继续由初始化 worker 和状态协调器维护。

不为 AICC 数据表增加 Pod 生命周期枚举，也不把运行时状态写回智能体的 `status`，避免 `active-but-starting` 等组合状态膨胀和双写漂移。

## 展示状态

后端为每个智能体统一计算面向页面的展示状态，前端不自行拼接业务规则。

| 展示状态 | 判定规则 | 页面行为 |
| --- | --- | --- |
| `starting`（启动中） | 隐藏 App 仍处于初始化阶段，或 `runtime_phase` 不为 `ready` | 禁用开始接待，显示运行时启动提示 |
| `ready`（待接待） | 运行时已就绪且智能体状态为 `draft` | 可开始接待 |
| `receiving`（接待中） | 运行时已就绪且智能体状态为 `active` | 公开页与挂件可接待访客 |
| `paused`（已暂停） | 运行时已就绪且智能体状态为 `paused` | 可恢复接待，公开入口拒绝新会话 |
| `error`（异常） | 隐藏 App 处于 `error` 或无法提供可用运行时 | 显示错误摘要和既有重试入口 |
| `deleted`（已删除） | 智能体状态为 `deleted` | 不提供操作入口 |

运行时已经就绪后，`draft` 的用户可见名称统一改为“待接待”。“启动中”只用于运行时尚未准备完成的过渡过程。

## 数据与接口

1. AICC 查询在读取智能体时，根据 `app_id` 查询其隐藏 App 的 `status`、`runtime_phase` 与安全的错误摘要。
2. AICC 响应增加只读运行时展示字段，例如 `runtime_status` 与 `runtime_message`；不暴露容器坐标、令牌或原始运行时快照。
3. 列表和详情使用同一服务层映射，保证顶部选择器、接待台、设置页与公开入口的状态口径一致。
4. 既有 `start` / `stop` 接口仍只更新智能体接待意图；运行时初始化、重启和故障收敛仍由现有 App worker 与 reconciler 负责。

## 前端交互

- 顶部当前智能体区域与接待台展示统一的运行时状态标签。
- “开始接待”仅在 `ready` 或 `paused` 时可用；`starting` 时显示等待提示，`error` 时按现有可恢复能力提供重试。
- 运行时状态存在轮询窗口时，复用 AICC 智能体查询刷新机制，直至到达稳定状态。
- 公开页继续仅以智能体接待意图 `active` 为准；运行时未就绪时服务端返回可识别的不可用错误，前端展示“客服正在启动，请稍后再试”。

## 测试与验证

- 服务层单元测试覆盖初始化中、就绪待接待、就绪接待中、暂停、异常、删除六种映射。
- Handler 与 OpenAPI 测试覆盖新增只读字段及权限隔离。
- 前端单元测试覆盖状态文案、按钮启用条件与状态轮询。
- 使用本地 k3d 真实浏览器验证：新建智能体显示“启动中”，Pod 就绪后变为“待接待”，点击开始接待后变为“接待中”，暂停后变为“已暂停”。
