# AICC 生产就绪测试设计

- 日期：2026-07-12
- 状态：已确认
- 验证环境：本地 k3d
- 容量目标：100 个并发访客，持续 30 分钟

## 1. 目标与结论边界

本设计用于严格验证 AICC 在线客服子系统是否达到上线门禁。测试覆盖功能、权限、安全、故障恢复、容量、升级和回滚，并为每项结论保留本轮可复核证据。

通过本设计只能证明当前代码在生产同构程度有限的本地 k3d 环境达到功能和稳定性门禁。本地容量结果不能替代真实生产基础设施的容量评估；若生产环境的节点规格、网络、存储或外部服务配额不同，上线前仍需核对这些差异。

## 2. 测试基线

执行测试前，先将在线客服需求文档同步为以下最终业务规则：

- 企业管理员从概览进入独立的 `/aicc-console` 工作台；平台管理员从企业列表进入指定企业的 AICC 工作台。
- 公开页不提供单条助手回复反馈，改用会话级“已解决”和“未解决”。
- 用户可见状态统一为“跟进中”“已解决”“未解决”，不展示历史实现词汇或“未知”。
- 公开页和挂件打开时不创建空 session，首次发送消息时才创建；刷新恢复原会话，点击“新建对话”建立新会话边界。
- 当前客服知识库始终参与检索；企业知识库可选择是否使用；行业知识库只能从平台为企业授权的范围中选择。
- 当前支持公开客服界面国际化，不支持语音客服。
- GeoIP XDB 数据随镜像构建，并支持运行期定期更新。

建立“需求条目 -> 自动化用例 -> 浏览器场景 -> 验证证据”的覆盖矩阵。每项结果只能为：

- `PASS`：存在本轮新鲜、可复核的通过证据。
- `FAIL`：行为不符合基线，需要修复并重新验证。
- `BLOCKED`：受环境或外部条件限制，不能计为通过。
- `N/A`：明确不在本期范围，并记录依据。

不得使用历史测试结果、相似功能测试或仅有实现代码代替本轮证据。

## 3. 分层上线门禁

测试按以下顺序执行。前一层通过后才进入下一层；任何关键失败均进入缺陷闭环。

### 3.1 静态、契约与构建

- 运行 Go 全仓测试、前端单元测试和现有端到端测试。
- 运行 TypeScript 类型检查、前端生产构建和 OpenAPI 同步检查。
- 检查 Git 工作区，确保没有无关改动、测试令牌、密码、临时调试代码或测试产物。
- 构建最终 manager-api、manager-web 和 Hermes runtime 镜像，验证镜像内容和启动健康状态。
- 扫描仓库与镜像配置中的敏感信息，并核对生产禁止项未被启用。

### 3.2 后端、数据与权限

- 智能体新建、编辑、启停、删除和数量上限。
- 平台开通/关闭 AICC、行业知识库授权和企业边界。
- 客服库、企业库和行业库的选择、检索与取消选择后的行为。
- session 延迟创建、恢复、新建对话、消息上限和零消息过滤。
- 会话三态、筛选、分页、来源、地域和消息详情。
- 留资字段、留资恢复、线索合并、已读状态、关联会话和 CSV 导出。
- 统计时间范围、趋势、地域、来源、问题、线索和未解决率口径。
- 访客限流、封禁、域名白名单、敏感词和余额不足路径。
- 数据保留调度、GeoIP 内置库和运行期更新。
- platform_admin、org_admin、org_member、匿名访客四类身份的允许和拒绝路径，包含跨企业访问。

### 3.3 真实浏览器

使用真实 Chrome，以用户可见行为为准覆盖公开页、挂件和工作台。每个场景同时检查页面状态、网络请求、浏览器控制台和后端日志。

浏览器覆盖桌面视口和移动端 Chrome 视口。页面不得出现控件遮挡、文本溢出、菜单错位、状态不同步或无法操作。

### 3.4 安全

- 未登录、普通成员、跨企业和伪造 token 访问必须被拒绝。
- 域名白名单必须阻止未授权站点使用挂件。
- 提示词注入不得泄露系统提示词、突破知识范围或触发非客服操作。
- 上传文件必须校验格式、大小、空文件和恶意文件名。
- 消息数量限制、频率限制和访客封禁必须真实生效，并提供可理解的错误提示。
- 敏感词和有害内容路径必须按配置处理。
- 组织、智能体、session、消息、线索和知识库数据不得跨租户读取或写入。

### 3.5 故障恢复

在当前 k3d 环境依次注入以下故障：

- Hermes 实例停止或不可达。
- RAGFlow 停止、超时或返回错误。
- new-api 停止、余额不足或模型调用失败。
- Redis 停止并恢复。
- MySQL 短时不可用并恢复。
- manager-api Pod 重启和滚动更新。

每个故障验证错误提示、超时边界、重试行为、数据一致性、恢复后续聊，以及重复请求不会产生重复 session、消息或线索。故障注入后必须恢复环境并执行核心冒烟回归。

### 3.6 容量与稳定性

容量门禁为 100 个并发访客持续 30 分钟，并同时记录平台接口耗时和完整模型回答耗时。

通过标准：

- 消息请求成功率不低于 99.5%。
- 完整消息请求 P95 不高于 15 秒，同时记录 P50 和 P99。
- 测试期间无服务进程或 Pod 异常重启。
- 不发生跨 session、跨智能体或跨企业的数据串写。
- 错误率、连接数、内存、CPU、数据库连接和队列积压在测试结束后回落。
- 所有失败请求均能按错误类型解释，不允许存在持续增长的未知错误。

若本地机器资源成为瓶颈，结果记为 `BLOCKED`，不得降低标准后宣称通过。

### 3.7 升级与回滚

允许清空当前 k3d 业务数据。升级测试按以下路径执行：

1. 使用 `master` 基线代码、数据库和镜像启动环境并生成固定验收数据。
2. 切换到 AICC 最终代码和镜像，执行数据库迁移并验证历史数据、权限和核心业务。
3. 回滚应用镜像，验证数据库兼容边界和服务可恢复性。
4. 恢复最终镜像，再次运行迁移、健康检查和核心浏览器冒烟测试。

如迁移不可重复、历史数据丢失、旧版本无法按预期回滚，或恢复最终版本后业务异常，均判定上线门禁失败。

## 4. 浏览器业务矩阵

### 4.1 平台管理员

- 创建和编辑企业时开通/关闭 AICC。
- 设置智能体数量上限和行业知识库授权范围。
- 从企业列表进入指定企业工作台。
- 跨企业查看时仅具备设计允许的只读能力。
- 关闭 AICC 后，企业入口消失，管理端访问被拒绝，公开链接和挂件立即下线。

### 4.2 企业管理员

- 从概览子系统入口进入独立工作台。
- 新建、编辑、切换、启停和删除智能体。
- 验证设置表单、字段说明、保存反馈和页面滚动稳定性。
- 配置知识范围和留资字段。
- 在当前客服知识库页面上传、解析、搜索、分页、下载和删除文档。
- 生成并预览公开链接、二维码和网页挂件。
- 查看会话、线索、关联对话、统计并导出 CSV。

### 4.3 企业普通成员

- 概览不展示 AICC 入口。
- 直接访问工作台路由被拒绝。
- 调用所有 AICC 管理 API 被拒绝。
- 不得通过已知 org_id 或 agent_id 越权读取数据。

### 4.4 匿名访客

- 打开公开页或挂件不创建 session，也不产生零消息会话。
- 首次发送消息时创建 session，回答成功并记录正确来源和地域。
- 刷新后恢复原 session 和消息；点击“新建对话”后使用新的 session。
- 隐私提示只在首次进入时展示，不持续占用消息区。
- 会话级“已解决”“未解决”与后台状态、筛选和统计一致。
- 已完成留资的恢复会话不再次强制弹出留资表单。
- 消息上限、频率限制、敏感词、封禁和服务不可用提示正确。
- 中文和英文界面中的用户可见文案、表单和错误提示完整。

### 4.5 知识问答

- 分别上传并查询客服库、企业库和每个已授权行业库的唯一验证口令。
- 禁用企业库或取消行业库后，对应口令不再命中。
- 当前客服知识库始终命中，不提供单文档启停概念。
- 文档解析失败、RAGFlow 不可用和无匹配知识时均有稳定行为。
- 提示词注入不能绕过知识范围、读取其他企业资料或调用非客服能力。

二维码只验证可正确解码为公开链接，不依赖微信扫码。消息通道只覆盖本期支持的公开链接和网页挂件，不扩展到语音或外部即时通信渠道。

## 5. 缺陷闭环

发现问题后执行以下闭环：

1. 保存截图、网络请求、服务日志、测试输入和复现步骤。
2. 能自动化复现时，先补充失败用例，再修复实现。
3. 运行改动相关测试，并按业务阶段使用中文 Conventional Commits 单独提交。
4. 重新运行当前层和已通过的上游回归。
5. 所有问题修复后，从干净数据重新执行整套门禁。

不能将修复前后多轮零散结果拼接为全量通过证据。

## 6. GO / NO-GO 标准

满足以下全部条件才可给出 `GO`：

- 覆盖矩阵中没有 `FAIL`，没有影响上线判断的 `BLOCKED`。
- 单元、API、组件和 Playwright E2E 全部通过。
- 四类身份的浏览器矩阵全部通过。
- 无 P0/P1 缺陷，无跨租户、消息丢失、重复计费或数据不一致。
- 安全、故障恢复、容量、升级和回滚门禁全部通过。
- 最终镜像、数据库版本、配置和测试报告可追溯到同一 Git 提交。
- 工作区干净，没有密钥、临时数据或未提交修复。

出现以下任一情况必须给出 `NO-GO`：

- 自动化测试失败或关键场景只能人工绕过。
- 存在跨租户访问、会话串数据、消息丢失或重复扣费。
- 依赖恢复后无法继续服务，或迁移/回滚导致数据损坏。
- 容量指标未达标，或者测试因资源限制无法完成。
- 最终运行镜像与已测试提交不一致。

## 7. 测试报告

最终报告至少包含：

- 需求覆盖矩阵和每项结果。
- 自动化命令、用例数量、通过率和失败证据。
- 浏览器覆盖的身份、页面、语言和视口。
- 容量测试的成功率、P50/P95/P99、错误分类和资源曲线。
- 故障注入、恢复时间和数据一致性结果。
- 升级、迁移和回滚结果。
- 已修复缺陷及对应提交。
- 剩余风险和最终 `GO / NO-GO` 结论。

测试产物不得保留真实令牌、密码或访客隐私数据。测试完成后重新生成固定的本地验收账号和数据。
