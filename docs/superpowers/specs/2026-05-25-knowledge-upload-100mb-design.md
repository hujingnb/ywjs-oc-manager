# 设计文档：知识库上传单文件上限统一为 100MB

**日期：** 2026-05-25
**状态：** 待用户复核

## 背景

组织成员在实例详情页的“实例知识库”上传文件时遇到 HTTP 413。当前链路存在两处不一致的上限：

- 生产入口 `deploy/manage/nginx.conf` 配置 `client_max_body_size 32M`，超过约 32MB 的请求会在 nginx 层直接返回 413，manager-api 不会收到请求。
- manager 知识库主副本通过 `files.NewSafeRoot(..., 0)` 使用默认单文件上限 50MB，超过该值会在 Go 服务层拒绝写入。

用户确认新的业务要求是：知识库上传单文件最多 100MB，并在前端给出明确提示。

## 目标

- 组织知识库和实例知识库上传单文件上限统一为 100MB。
- nginx、manager-api 后端校验、前端提示和前端本地拦截保持一致。
- 用户选择超过 100MB 的文件时，前端直接提示并中止上传，避免等待网络请求失败。
- 保留现有上传进度、取消上传、权限判断和同步入队逻辑。

## 非目标

- 不做分片上传、断点续传或刷新后恢复。
- 不改变知识库目录结构、同步任务、权限规则或数据库 schema。
- 不扩大其他非知识库上传能力；例如 assistant skill 上传是否调整不在本次范围内。

## 推荐方案

采用“全链路统一上限”的小范围修改：

1. 部署层把 `deploy/manage/nginx.conf` 的 `client_max_body_size` 从 `32M` 调整为 `100M`。
2. 后端装配知识库主副本时显式传入 `100 * 1024 * 1024`，避免继续依赖 `SafeRoot` 的 50MB 默认值。
3. 前端知识库上传入口共享一个 `100MB` 常量，用于提示文案和选择文件后的本地校验。
4. 实例知识库和组织知识库都展示同样限制，避免同一后端能力在两个页面表现不一致。

该方案比只改 nginx 或只改前端更稳妥：超过 32MB 的请求能进入服务，超过 50MB 但不超过 100MB 的文件能被后端接受，超过 100MB 的文件会在前端和后端两侧都被拒绝。

## 改动范围

### 后端与部署

- `cmd/server/main.go`
  - 将知识库 `SafeRoot` 初始化从默认上限改为显式 100MB。
  - 建议新增局部常量或复用文件包常量，注释说明这是知识库业务单文件上限。
- `internal/files/safe_path.go`
  - 若实现需要复用常量，可新增导出的 `DefaultMaxFileSize` 或 `KnowledgeMaxFileSize`。
  - 不改变路径安全校验语义。
- `deploy/manage/nginx.conf`
  - 将 `client_max_body_size 32M;` 调整为 `client_max_body_size 100M;`。

### 前端

- `web/src/api/hooks/useKnowledge.ts` 或新增小工具文件
  - 定义知识库上传上限常量：`100 * 1024 * 1024`。
  - 导出格式化后的提示文案，供组织知识库和实例知识库页面复用。
- `web/src/pages/apps/AppKnowledgeTab.vue`
  - 上传按钮附近展示“单文件最多 100MB”。
  - `onUploadFile` 读取 `file.size` 后先校验，超过上限则用 `n-message.warning` 提示并返回，不调用 `uploadProgress.run`。
- `web/src/pages/knowledge/OrgKnowledgePage.vue`
  - 同步添加提示与本地校验。

## 行为设计

### 正常路径

1. 用户在组织知识库或实例知识库选择文件。
2. 前端检查 `file.size <= 100MB`。
3. 合法文件进入现有 `uploadProgress.run` 流程。
4. `xhrUpload` 继续按当前逻辑展示进度、支持取消、携带鉴权与 CSRF 头。
5. 后端写入知识库主副本并触发已有同步分发逻辑。

### 超限路径

1. 用户选择超过 100MB 的文件。
2. 前端清空 file input，展示“单文件最多支持 100MB”的提示。
3. 不创建上传进度会话，不发送 HTTP 请求。
4. 如果绕过前端直接调用 API，后端仍通过 `SafeRoot` 上限拒绝写入。

### 网关路径

生产入口允许最大 100MB 请求体。超过该值的请求仍可能由 nginx 返回 413，这是预期行为；正常浏览器入口会先被前端拦截。

## 测试计划

### Go 单元测试

- 更新 `internal/files/safe_path_test.go` 或相关装配测试，覆盖默认/显式上限语义。
- 更新 `internal/files/knowledge_master_test.go`，覆盖 100MB 上限附近的拒绝逻辑时使用小上限注入，不实际构造 100MB 大文件。

### 前端单元测试

- `AppKnowledgeTab`：选择超过 100MB 的文件时提示 warning，且不调用上传 mutation / `uploadProgress.run`。
- `OrgKnowledgePage`：同样覆盖超限文件不发起上传。
- 已有上传成功路径保持不变，确认合法文件仍会进入现有上传进度流程。

### 手动验证

- 用真实浏览器登录组织成员账号，进入实例详情页的实例知识库。
- 上传小于 100MB 的文件，确认进度弹窗正常、上传成功、列表刷新。
- 选择大于 100MB 的文件，确认前端立即提示且无网络上传请求。
- 如本地环境可模拟 nginx，确认 32MB 到 100MB 的文件不再被 nginx 413 拦截。

## 风险与兼容性

- 将 nginx 和后端上限提高到 100MB 会增加单次请求占用的网络和磁盘写入时间；当前已有上传进度与刷新确认，可缓解用户误操作。
- 不做分片上传意味着弱网下大文件失败仍需重传，这是本次明确不解决的限制。
- 如果生产还有额外反向代理或云网关，它们也需要同步配置到不低于 100MB，否则仍可能在更外层返回 413。
