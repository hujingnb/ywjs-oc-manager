# 用量统计页面数据修复设计

## 背景

用量统计页面在平台管理员、组织管理员和组织成员身份下都没有数据展示。现有架构要求
manager 不缓存用量数据，而是通过 `UsageService` 直查 `new-api`，前端再按平台、组织、
成员和应用维度展示结果。

## 目标

- 修复用量页没有数据的问题，覆盖平台、组织和成员三个主要入口。
- 保持现有 API 契约和薄代理语义，不新增本地用量缓存。
- 使用浏览器验证不同身份下的展示和权限裁剪是否正常。

## 设计

### 前端查询边界

`UsagePage.vue` 继续负责按角色裁剪 tab 和查询入口。普通组织成员的“我的用量”必须使用
当前用户所属组织作为权限查询参数，不能复用被组织维度禁用的 `orgRef`，否则 TanStack Query
不会发起成员用量请求。

平台管理员和组织管理员保持现有组织维度查询入口。平台管理员首次加载组织列表后仍默认选中
第一个组织，避免组织维度出现无目标资源的空查询。

### 后端时间窗口

组织和平台维度在用户没有显式传入 `since` / `until` 时，后端提供最近 30 天的默认窗口。
这样既符合“本月 / 近期用量”的页面语义，也避免把 `0,0` 直接传给 `new-api` 后在部分部署中
返回空数据。

成员和应用维度仍走 `new-api` 日志端点，保留现有分页默认值。若没有绑定应用或 api key，
继续返回空列表，这代表真实无用量来源。

### 展示与权限

`UsageSummary.vue` 保留动态列展示方式。修复后页面应能区分三种情况：

- 请求成功且有 `items`：展示记录数和表格。
- 请求成功但无 `items`：展示对应维度的空态。
- 请求失败：展示后端错误信息，浏览器 Network 中不应出现因前端参数缺失导致的无谓 400/403。

权限仍由后端最终判断。前端只做入口裁剪：平台管理员可见平台、组织、成员和应用入口；组织管理员
可见组织、成员和应用入口；组织成员只看“我的用量”和应用入口。

## 测试计划

- 后端单元测试覆盖默认时间窗口和成员自查参数路径。
- 如前端测试框架能低成本覆盖，则补充 `UsagePage` 或 hook 测试，确保成员身份会发起成员用量请求。
- 浏览器验证：
  - 先打开当前用量页，确认修复前/后实际是否有数据展示。
  - 使用平台管理员、组织管理员、组织成员分别登录后访问 `/usage`。
  - 检查不同身份可见 tab 是否符合权限裁剪。
  - 检查 Console 无前端异常，Network 中用量接口无非预期 400/403/500。

## 非目标

- 不实现新的日期筛选 UI。
- 不改变 `openapi/openapi.yaml` 和前端生成类型，除非实际 handler 契约发生变化。
- 不增加 manager 本地用量缓存或聚合表。
