# 全量中文注释补齐设计

## 背景

当前仓库已有中文注释规范：新增或修改代码注释必须使用中文，并且相关逻辑需要覆盖文件、方法、结构体、字段、常量、参数和关键代码段。现有代码中部分核心文件已经具备较完整的中文注释，但不同模块覆盖程度不一致，尤其是测试辅助类型、前端组合逻辑、局部业务分支和部分字段语义仍存在缺口。

本次任务目标是在不改变业务行为的前提下，补齐所有人工维护源码中缺失或不足的中文注释，让读者可以从注释理解业务意图、权限边界、状态语义、异常原因和非显而易见的实现约束。

## 范围

纳入范围：

- 后端 Go 生产代码：`internal/**/*.go`、`cmd/**/*.go` 等人工维护文件。
- 后端 Go 测试代码：`*_test.go` 中的测试场景、fake/mock、关键断言意图。
- 前端 TypeScript：`web/src/**/*.ts` 中的 API、domain、store、router、composable 等人工维护代码。
- 前端 Vue 单文件组件：`web/src/**/*.vue` 中的业务计算、事件处理、权限判断、数据流和关键模板分支。

排除范围：

- 生成产物：`openapi/openapi.yaml`、`web/src/api/generated.ts`。
- 依赖、构建产物和临时目录：`node_modules`、`dist`、`coverage`、`tmp` 等。
- 任何带明确生成标记且由工具维护的文件。

## 注释标准

注释必须使用中文，保留第三方接口名、协议名、错误码、命令、配置键和代码标识的英文原文。

文件级注释说明该文件在模块中的职责和边界。Go 文件优先使用 package 或文件顶部注释；Vue/TypeScript 文件在脚本区域顶部补充职责说明。

类型、接口、结构体和字段注释说明对外语义、数据来源、权限含义、生命周期或空值边界。对于仅在测试中使用的 fake/mock，也说明其模拟的依赖能力和测试约束。

函数和方法注释说明业务目的、关键参数含义、主要返回结果和重要错误边界。参数语义明显时不强行逐参数堆砌重复描述，但存在权限、状态、路径、安全或数据转换含义的参数必须解释。

常量和枚举注释说明业务含义、与后端/前端契约的对应关系、状态机约束或外部系统约定。

关键代码段注释放在复杂分支、权限判断、事务边界、重试/降级逻辑、状态转换、路径安全处理、跨系统调用、异常吞吐和测试断言组之前。注释应解释“为什么这样做”，避免重复“代码正在做什么”的表面含义。

不添加低价值注释，例如简单赋值、显而易见的返回语句、重复表驱动 case 的逐项描述。若某段代码已经有清晰中文注释，仅在确实缺少边界或意图时补充。

## 执行策略

先生成候选清单，再按模块人工补齐，避免机械批量插入低质量注释。

处理顺序：

1. 后端核心契约与权限：`internal/auth`、`internal/domain`、`internal/config`。
2. 后端业务服务与 API：`internal/service`、`internal/api`。
3. 后端异步与集成：`internal/worker`、`internal/scheduler`、`internal/integrations`、`internal/runtime`、`internal/files`、`internal/audit`、`internal/log`。
4. 后端存储与测试辅助：`internal/store` 及相关 `*_test.go`。
5. 前端公共逻辑：`web/src/api`（排除生成文件）、`web/src/domain`、`web/src/stores`、`web/src/composables`、`web/src/app`。
6. 前端页面与组件：`web/src/pages`、`web/src/components`、`web/src/layouts`、`web/src/App.vue`。

每个模块只修改注释，不改行为、不重排无关代码、不做无关格式化。若补注释时发现已有注释与实际行为不一致，优先修正注释；只有发现明确 bug 时记录给用户，不在本任务中扩展修复范围。

## 验证

后端文件修改后运行 `gofmt`。如果只改注释，仍运行与被触达模块相关的 Go 测试；若改动范围过大，优先运行 `go test ./internal/...`，必要时再补充更窄测试定位。

前端文件修改后运行项目既有检查命令，优先从 `package.json` 中选择类型检查、测试或构建脚本。若环境依赖缺失或命令不可用，在交付说明中列出实际命令、失败原因和风险。

交付前检查工作区，确认没有修改生成产物、没有混入无关文件、没有引入调试代码或密钥信息。

## 交付物

最终交付应包含：

- 补齐中文注释后的人工维护源码。
- 已运行的验证命令和结果。
- 若有无法覆盖的文件或无法运行的检查，说明原因。
