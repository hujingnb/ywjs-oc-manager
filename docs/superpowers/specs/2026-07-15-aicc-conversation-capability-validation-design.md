# AICC 客服对话、能力沙箱与全面验证设计

- 日期：2026-07-15
- 状态：设计确认稿
- 关联设计：`2026-07-14-aicc-stateless-runtime-app-type-design.md`

## 1. 背景

现有 AICC 已具备公开会话、企业/行业/客服知识库、隐私同意、留资字段、联系人线索、
异步回复、限流、统计和真实浏览器 E2E。当前实现仍沿用通用 Hermes runtime 的若干能力：

- AICC 平台提示词要求检查已安装 Skill；
- 知识库检索通过 terminal 执行 `oc-kb search`；
- runtime 仍注册本地 terminal，并关闭危险命令审批；
- 通用 memory 和 user profile 仍被启用；
- 必填留资字段在首次咨询前阻断对话；
- 单条回复反馈直接覆盖整场会话的解决状态；
- 线索依赖显式联系方式，尚无自然对话意向画像；
- AICC 本地 Hermes session 仍可能被当作多轮上下文来源。

这些行为与“客服只能问答、指导和提取线索，不能执行操作”的产品边界冲突，也不能满足
无状态多副本 runtime 的恢复要求。

## 2. 产品目标

本设计的目标是：

1. 保留 Hermes 的多轮对话、自主工具选择和 Skill 生态。
2. 为 AICC 建立默认拒绝的能力沙箱，首期只允许知识库检索和只读网络搜索。
3. 优先依据企业知识回答；知识不足时允许用公开网络补充，但必须区分未经企业确认的信息。
4. 客服可以解释企业产品的正常使用步骤，但不能代替访客执行任何操作。
5. 取消首次咨询前的留资阻断，从自然对话中识别并沉淀可解释的购买意向。
6. AICC runtime 完全无状态、无 S3 运行时持久化，并支持幂等启动和任意副本续聊。
7. 用全场景可追溯矩阵证明所有已识别业务分区、状态、权限和失败模式均已验证。
8. 核心对话链路保持渠道无关，为后续语音客服接入预留稳定适配边界。

## 3. 非目标

本期不实现：

- 创建订单、工单、预约或 CRM 写入等业务操作；
- 让企业管理员放宽平台 capability 上限；
- 企业自定义可执行 Skill 或运行时动态创建 Skill；
- 跨访客长期记忆或匿名访客画像合并；
- 自动修改知识库；
- 单条助手回复的“有帮助 / 没帮助”评价及回答质检工作台；
- ASR、TTS、电话线路、实时音频传输或具体语音服务商接入；
- 生产集群验证或生产数据操作。

未来若开放写操作，必须设计新的授权等级、访客确认和审计闭环，不能直接放宽本期公开客服
策略。

## 4. 总体架构

公开消息主链路保持：

```text
访客
  → manager AICC 公开 API
  → 消息持久化与异步任务
  → Hermes AICC runtime
  → 镜像内 Capability Broker
  → manager 只读 runtime API / 受控网络搜索出口
  → manager 持久化回复、来源、意向和审计
```

职责边界：

- manager 是会话、消息、意向、联系人线索、邀请状态和解决状态的唯一真相源。
- Hermes 负责多轮理解、允许范围内的 Skill 自主选择和自然语言回复。
- Capability Broker 内置于 `hermes-aicc` 镜像，不新增独立微服务。
- manager 对 runtime API 再做服务端授权，不能只信任容器内 Broker。
- RAGFlow 提供企业、行业和当前客服知识检索；受控搜索出口提供公开网络只读能力。
- 网页、挂件和未来语音入口通过 channel adapter 归一化为同一对话轮次，不把渠道逻辑写进
  Hermes、Capability Broker 或意向分析器。

## 5. Capability Broker

### 5.1 权限模型

每个客服 Skill 声明所需 capability。最终权限取三者交集：

```text
镜像固化上限 ∩ 企业启用项 ∩ Skill 声明 = 实际可用能力
```

首期允许：

- `knowledge.read`
- `web.search`
- 对话回复
- 隔离上下文中的结构化意向分析

首期明确拒绝：

- `terminal.execute`
- 文件读取和写入
- 代码执行与进程启动
- 应用创建、建站和部署
- 网络写请求、登录和表单提交
- 凭证访问
- 未声明或未审核的能力

### 5.2 三层防线

1. 启动阶段只安装或呈现客服白名单 Skill；普通 Hermes Skill 不自动进入 AICC。
2. AICC 不注册通用 terminal、文件和进程工具；知识检索与网络搜索使用专用只读工具。
3. 每次调用按 app type、Skill 声明、镜像上限和请求内容执行时授权。

manager runtime API 使用 app-scoped token 再次校验只读权限。Pod 网络策略只允许访问必要的
manager API、模型网关和受控搜索出口，并拒绝集群内其他服务、云元数据地址和任意写出口。

### 5.3 Skill 生命周期

首期内置三个版本受控的客服 Skill：

- `customer-answer`：来源优先级、客服边界、操作拒绝和自然表达。
- `safe-web-research`：只读搜索、来源筛选和企业信息标记。
- `lead-analysis`：意向画像、证据和一次性邀请策略。

这些 Skill 构建进不可变 AICC 镜像，bootstrap manifest 只能选择其子集。新增平台客服 Skill
通过新镜像版本发布，不在运行时下载、创建或修改。

## 6. 回答与来源策略

每轮访客消息先区分企业问答、企业产品指导、商务咨询、操作请求、无关任务和安全攻击。

正常企业咨询先执行 `knowledge.read`，检索顺序为：

```text
当前客服知识 → 企业知识 → 已授权行业知识
```

规则如下：

- 知识充分时依据知识库回答。
- 知识不足时，Hermes 可以自主调用 `web.search`。
- 网络补充通用信息时标明公开来源并提供可访问链接。
- 网络补充企业相关信息时必须标明“来自公开网络，未经企业确认”。
- 网络与企业知识冲突时以企业知识为客服口径；差异影响访客决策时主动说明存在冲突。
- 没有可靠依据时明确无法确认，不补全价格、政策、时效、资格或结果承诺。
- 保存知识命中标识、网页 URL、标题、检索时间和关联消息，供后台审计。

客服可以解释企业产品的正常使用步骤，但不得替访客登录、提交、上传、执行或修改。创建
网站、运行命令、修改文件、启动进程等非客服任务必须简短拒绝并引导回企业业务范围。

回复展示前校验是否声称执行操作、泄露内部实现、混淆来源、遗漏网络标记或作出无证据
承诺。校验失败时内部重写一次，仍失败则返回安全兜底。

## 7. 意向识别与留资

### 7.1 非阻断体验

访客在隐私政策允许后立即咨询。原 `required` 语义改为“访客决定提交表单时该字段必填”，
不再阻断首次咨询。

意向等级定义：

- `low`：泛泛了解，无具体购买或落地需求。
- `medium`：存在明确场景、产品兴趣或对比需求，但尚无推进计划。
- `high`：询价、试用、演示、合同、预算、采购时间，或明确要求销售跟进。

提取字段包括关注产品、需求场景、预算、采购时间、企业/行业、决策角色、主要顾虑和
联系方式。每个结论必须引用当前 session 的访客 `message_id`，没有证据时保持为空。
显式填写优先于模型推断；不得推断年龄、性别、收入等无关敏感属性。

### 7.2 分析与邀请

`lead-analysis` 在公开消息异步任务中、Hermes 生成访客回复前运行。它使用隔离分析上下文，
输出严格 JSON，不直接写数据库。manager 校验字段、证据归属、置信度、分析版本和幂等键后
持久化，并将本轮邀请策略传给 Hermes。

邀请规则：

- 高意向且本 session 从未邀请过时，自然邀请一次并展示可选留资卡片。
- 访客明确拒绝后标记 `declined`，本 session 不再追问。
- 访客忽略邀请可继续咨询，不重复展示。
- 提交后标记 `submitted`。
- 中低意向不主动索要联系方式。
- 意向分析失败不阻断回复，也不错误邀请；后台记录并幂等重试。

### 7.3 数据分层

新增 `aicc_session_intents`，每个 session 唯一一行，记录意向等级、结构化字段 JSON、
字段级置信度、证据 JSON、分析器版本、最近已分析访客消息和邀请状态。邀请状态固定为
`not_invited`、`invited`、`declined` 或 `submitted`。高意向且尚未关联正式线索的
会话直接从该表派生为匿名意向候选，不再增加第二份候选数据。

现有 `aicc_leads` 继续表示有联系方式、可跟进的正式线索，并按企业和联系方式去重。匿名
候选后来提交联系方式时关联或合并到正式线索，一个联系人可关联多个会话画像。

## 8. 会话解决状态

移除公开端单条回复的“有帮助 / 没帮助”，`aicc_feedback` 不再决定 session 状态。

会话状态只由访客明确选择和新消息驱动：

```text
unknown → resolved
unknown → unresolved
resolved → unknown（同一 session 再次提问）
unresolved → unknown（同一 session 再次提问）
新建对话 → 新 session / unknown
```

`unknown` 表示进行中或未确认。第一次在第二条非拒答助手回复完成后，以非模态卡片询问
“这次咨询解决了吗？”。访客继续提问时立即收起；同一处理阶段不再自动弹出，之后仅在访客
点击“结束本次咨询”或发送明确结束语时再次展示。过期时仍为 `unknown` 的会话不自动推断
为未解决。

## 9. 访客交互

- 页头保留客服名称、在线状态、新建对话和次要样式的“结束本次咨询”入口。
- 输入框在隐私 gate 通过后立即可用。
- 回复阶段显示“正在查找企业资料”“正在核实公开信息”“正在组织回复”等业务提示，
  不暴露 Hermes、RAGFlow 或内部工具。
- 企业知识和公开网络以可点击来源标签展示。
- 企业相关网络信息显示醒目的“未经企业确认”标记。
- 留资卡片出现在高意向相关回复后，可填写或选择“暂时不用”。
- 解决确认是 session 级操作，不附着在每条助手回复上。

## 10. 客服专用 AICC 镜像

保留：

- Hermes 多轮对话、模型接入和 Skill 自主选择内核；
- 当前 session 内短期上下文；
- 客服白名单 Skill；
- 专用知识检索、网络搜索、健康检查、指标和结构化审计；
- 由 manager 校验后传入的当前轮图片理解。

移除或禁用：

- terminal、文件读写、workspace 交付；
- `oc-kb add`、`oc-publish`；
- cron、kanban、建站、进程管理；
- 通用频道登录和账号操作；
- 跨 session memory、user profile 和共享访客画像；
- 普通实例的工作目录、交付物和危险命令审批说明。

摘要、工具缓存和 Skill 临时状态必须按 session 隔离。正式联系人和意向数据只保存在
manager，不回灌为全局 Hermes 记忆。

## 11. 渠道无关接口与语音扩展

核心链路使用渠道无关的输入轮次和响应信封。manager 内部定义稳定的 channel adapter
边界，将 `web_link`、`web_widget` 和未来 `voice` 归一化为：

- session、channel 和当前消息标识；
- 已校验的文本或 ASR 转写文本；
- 语言、发生时间和可选媒体引用；
- 近期消息、受控摘要和当前策略。

Hermes 返回的响应信封包含：

- 可展示或播报的回答文本；
- 结构化来源；
- `none`、`offer_lead` 或 `ask_resolution` 等下一步动作；
- 拒答、兜底和来源校验结果；
- 本轮工具与安全审计引用。

网页 adapter 将下一步动作渲染为留资卡片或解决确认；未来语音 adapter 可将同一动作转换为
口头询问、DTMF 或通话平台事件。意向分析器只输出业务决策，不生成网页组件或电话供应商
指令。

语音接入时，ASR、TTS、呼叫生命周期、打断和音频传输位于独立 voice adapter；Hermes 和
Capability Broker 继续处理文本与结构化动作。语音渠道不能因此获得额外 capability，
隐私同意、来源政策、一次性留资邀请、session 解决状态和租户隔离继续复用现有规则。

现有 `aicc_sessions.channel='voice'` 枚举作为渠道标识保留。本期只建立接口、响应结构和
mock adapter 契约测试，不新增音频字段、供应商配置或空实现服务，避免为未确定的电话平台
提前固化错误模型。

## 12. 无状态与幂等启动

AICC Pod 的 `/opt/oc-input` 和 `/opt/data` 使用 `emptyDir`。容器不持有 S3 凭证，不执行
restore、sync 或 preStop 保存，也不把容器磁盘作为真相源。

启动顺序：

1. initContainer 只读获取 bootstrap manifest、persona、策略和 app-scoped token。
2. 专用 entrypoint 清理受管目录，在临时目录完整渲染配置。
3. 校验 manifest、内置 Skill 清单、capability 上限和禁止工具未注册。
4. 原子替换运行目录。
5. 启动 Capability Broker 和 Hermes。
6. Broker、Hermes 和策略校验均健康后才进入 Ready。

同一镜像和 manifest 重复执行必须得到等价结果。容器重启时即使 `emptyDir` 留有完整或
半成品目录，也走同一清理、暂存和原子替换路径。多个副本可并发 bootstrap，不创建共享
资源、不争抢启动锁。任一校验失败时失败关闭，不能回退为普通 Hermes 配置。

公开会话上下文由 manager 每轮提供：近期原始消息、受控摘要、当前问题、策略和幂等键。
任意全新 Pod 都能处理下一轮，不依赖本地 Hermes session。长会话摘要保存在 manager；
Pod 本地 session、缓存或临时文件丢失不影响续聊。

访客图片仍可由 manager 公共 API 保存到业务对象存储；AICC 容器不持有对象存储权限。

## 13. 错误处理

- 知识库失败：不得用网络结果冒充企业事实；可提供有来源的通用信息并说明企业资料暂不可用。
- 网络失败：知识充分时继续回答，否则说明公开信息暂无法核实。
- 意向分析失败：不阻断回复，不邀请；记录后幂等重试。
- Hermes 或模型超时：沿用异步队列、退避、熔断和手动重试。
- Broker 拒绝：作为确定性安全结果结束，不进入普通重试，并写安全审计。
- 来源校验失败：内部重写一次，仍失败则安全兜底。
- 重复提交：沿用客户端消息幂等键，不重复回复、邀请或创建线索。
- bootstrap 或策略校验失败：Pod 不 Ready，不使用空配置启动。

公开错误不得泄露服务名、内部路径、凭证、堆栈或治理阈值。

## 14. 全场景验证

### 14.1 可追溯要求

建立需求—状态—风险—测试矩阵。每条需求分配唯一 ID，并覆盖正常、拒绝、边界、依赖失败、
失败恢复、幂等、并发、多副本、权限、租户隔离、重启和浏览器交互。每条代码变更必须映射
到相应测试。

验证域完整覆盖：

1. 企业、行业、客服知识的单独和组合命中，无命中、模糊、过期、冲突和恶意文档。
2. 网络补充、来源、冲突、失败、不可信网页、SSRF、内网、云元数据和写请求。
3. 命令、代码、文件、进程、建站、登录、表单、写 API 及多轮/编码绕过。
4. Skill 安装、声明、越权、伪造、Broker 启动和执行时授权。
5. 多轮指代、追问、改口、话题切换、长摘要、Pod 重建和跨副本续聊。
6. 意向正例、反例、临界例、升级、降级、撤回、字段冲突和敏感属性边界。
7. 邀请、拒绝、忽略、提交、匿名转实名、去重、并发和事务回滚。
8. session 的全部合法与非法状态转换。
9. token、域名、隐私、限流、封禁、队列、图片、重放和跨 session 访问。
10. 全新/重复/残留/并发启动，非法 manifest、readiness 和无 S3 契约。
11. RAGFlow、搜索、new-api、Redis、数据库和 runtime 的失败、恢复、熔断与过载。
12. 独立页、挂件、桌面、移动端、中英文、刷新、重开、可访问性和控制台错误。
13. 普通 Hermes 的 terminal、workspace、S3、Skill、OpenAPI、迁移和历史数据回归。
14. channel adapter 的归一化、响应动作映射、未知渠道失败关闭，以及 mock voice adapter
    不改变 Capability、意向、隐私和 session 状态规则。

高风险组合全部覆盖，其余组合采用 pairwise；自由文本、URL、Skill manifest 和结构化输出
增加模糊或属性测试。不得用 skip、降低断言或删除有效测试交付。

### 14.2 验收门槛

- 需求矩阵映射率 100%。
- 已识别状态转换覆盖率 100%。
- capability 允许/拒绝决策覆盖率 100%。
- 依赖故障与恢复项通过率 100%。
- 关键浏览器旅程通过率 100%。
- 未授权操作成功次数为 0。
- 跨访客、跨客服、跨企业泄漏次数为 0。
- 企业事实有知识依据，或明确标为未经企业确认的网络信息。
- 意向证据全部引用当前 session 的真实访客消息。
- 高意向识别精确率不低于 90%，召回率不低于 85%。
- 重复留资邀请次数为 0。
- 所有相关失败项均修复并重新验证，不保留未处理的相关已知问题。

自然语言输入无法逐句穷举，但所有已识别业务分区、状态、权限、边界和失败模式必须通过
可追溯矩阵证明无遗漏。

### 14.3 浏览器标准

最终全量 E2E 使用 Playwright 驱动本地 Chrome Stable，配置 `channel: "chrome"` 并采用
headed 模式。关键旅程至少连续完整通过三轮，保存截图、视频、trace、控制台和网络证据。

另用人工 Chrome 复核视觉层级、动画节奏、中文输入法、移动端和挂件实际手感。curl 或直接
API 调用只能用于准备数据和辅助诊断，不能替代浏览器业务流程。

验证仅在本地 k3d 进行，不连接生产集群。

## 15. 交付顺序

1. 建立需求矩阵与现状基线。
2. 收紧 AICC 镜像、Capability Broker、专用工具和网络策略。
3. 完成无状态上下文、幂等启动和故障恢复。
4. 实现回答来源策略与输出校验。
5. 实现意向画像、非阻断邀请和匿名候选合并。
6. 建立渠道无关响应信封和 mock voice adapter 契约。
7. 修正 session 状态和访客交互。
8. 分层运行单元、集成、故障、模型评测和真实浏览器验证。
9. 对发现的问题先补复现用例再修复，直到全部硬门槛通过。
