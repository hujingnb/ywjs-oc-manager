# 定制技能工单交互重设计 设计文档

> 日期:2026-06-11
> 状态:设计已确认,待编写实现计划
> 关联:在 [[2026-06-10-定制技能-design]] 基础上重做工单工作流与界面

## 背景与目标

定制技能工单首版交互较糙,需重做。已确认的问题与目标:

1. **状态流转笨重**:平台侧用「下拉框 + 保存状态」改状态;且详情抽屉无视当前状态恒显示状态下拉,导致「已交付工单还能被打回待处理」。→ 改为**状态驱动的动作按钮**,状态由动作自动流转,不再手选。
2. **交付后不能改可见范围**:首次交付写死 targets,之后无任何编辑接口。→ 新增**交付后编辑可见范围**能力。
3. **详情页不美观、容器受限**:640px 抽屉 + 220px 右栏拥挤。→ 改为**独立整页详情**(平台侧与需求方共享),列表 **keep-alive 保活**(返回不刷新、保住滚动/筛选,变更后列表行自动更新)。
4. **对话/附件割裂**:评论与附件分两块,无时间戳。→ **统一消息流**:消息分 `text/image/file` 三类,附件不挂评论、自身即一条消息;每条带作者+发送时间;图片内嵌预览、文件下载条。
5. **双向改**:平台管理员侧与需求方(企业管理员/企业员工)侧**都**改成同一套整页详情 + 新对话。

范围:工单工作流这一个子系统(状态机校验 + 消息模型重构 + targets 更新接口 + 双侧整页详情 UI)。单个 spec。

## 现状关键事实(实现前必读)

- **状态机现状宽松**(`internal/service/skill_ticket_service.go`):`UpdateStatus` 仅允许 pending↔processing 但无当前状态前置校验;`Reject`/`Deliver` 对任意状态生效(rejected 也能交付);status 字段无 DB CHECK。
- **交付**(`internal/service/custom_skill_service.go:71-137`):`Deliver` 无前置状态校验;首次交付(`!ticket.CustomSkillName.Valid`)才写 `custom_skill_targets`,再次交付沿用既有 targets;交付后无 targets 更新路径(`DeleteCustomSkillTargetsByName` 已存在于 sqlc 但未接业务)。
- **对话/附件现状**:`skill_ticket_comments`(body TEXT)+ `skill_ticket_attachments`(object_path/file_name/file_size,comment_id 可空)两表;`AddComment` 在提交者于 delivered/rejected 评论时自动重开为 pending。
- **路由/权限**:工单相关路由见 `internal/api/handlers/skill_tickets.go`、`custom_skills.go`;`CanManageSkillTicket`=platform_admin、`CanViewSkillTicket`=platform_admin 或提交者本人、`CanSubmitSkillTicket`=org_admin/org_member(`internal/auth/authorizer.go`)。
- **前端现状**:平台侧 `web/src/pages/platform/CustomSkillTicketsPage.vue`(队列+宽抽屉+交付弹窗);需求方 `web/src/components/SkillTicketPanel.vue`(在 `OrgSkillsPage`→`SkillManager` 第三 tab);hooks 在 `web/src/api/hooks/useSkillTickets.ts`。
- **附件存储**:复用 `LibraryBlobStore`,现附件前缀 `library/ticket-attachment/`。
- 下一个迁移号:`000011`(确认 `internal/migrations/` 现最大为 000010)。

## 详细设计

### 1. 状态机(动作驱动 + 前置校验)

状态值不变:`pending | processing | delivered | rejected`。改为**显式转换动作 + 当前状态前置校验**,移除「任意手选状态」。

转换规则(service 层强校验当前状态,非法转换返回 `ErrInvalid`):

| 动作 | 谁 | 前置状态 | 结果状态 | 备注 |
|---|---|---|---|---|
| 开始制作 StartProcessing | platform_admin | pending | processing | |
| 交付 Deliver | platform_admin | pending **或** processing(首次)/ delivered(迭代) | delivered | rejected 禁止交付;首次写 targets,迭代沿用 |
| 拒绝 Reject | platform_admin | pending 或 processing | rejected | 写 reject_reason;delivered 不可拒绝 |
| 重新受理 ReopenRejected | platform_admin | rejected | processing | 管理员翻案,直接进制作中 |
| 设报价 SetQuote | platform_admin | pending 或 processing | (不变) | 「未交付阶段」可改;delivered/rejected 禁止 |
| 编辑可见范围 UpdateTargets | platform_admin | delivered | (不变) | 见 §3 |
| 发消息自动重开 | 提交者本人 | delivered 或 rejected | pending | 见 §2;管理员发消息不触发重开 |

- **移除**通用 `UpdateStatus(pending↔processing)`,改为上述显式方法。前端不再有状态下拉/保存。
- **不做**:制作中→退回待处理;手动设 delivered;DB CHECK 约束(沿用应用层校验风格)。

### 2. 统一消息流(改表,破坏性迁移)

**破坏性迁移 `000011`**(不兼容历史数据,用户已确认):`DROP TABLE skill_ticket_comments; DROP TABLE skill_ticket_attachments;` 并新建:

```sql
CREATE TABLE skill_ticket_messages (
    id             CHAR(36)    NOT NULL                 COMMENT '主键 UUID',
    ticket_id      CHAR(36)    NOT NULL                 COMMENT '所属工单',
    author_user_id CHAR(36)    NOT NULL                 COMMENT '发送者 user id(提交者或平台管理员)',
    kind           VARCHAR(16) NOT NULL                 COMMENT '消息类型:text|image|file,决定 body 的 JSON 结构与前端渲染',
    body           JSON        NOT NULL                 COMMENT '消息内容,按 kind 变化:text={"text":".."};image/file={"object_path":"..","file_name":"..","file_size":N,"content_type":".."}',
    created_at     DATETIME(6) NOT NULL DEFAULT CURRENT_TIMESTAMP(6) COMMENT '发送时间',
    PRIMARY KEY (id),
    CONSTRAINT fk_skill_ticket_messages_ticket FOREIGN KEY (ticket_id) REFERENCES skill_tickets(id),
    CONSTRAINT fk_skill_ticket_messages_author FOREIGN KEY (author_user_id) REFERENCES users(id),
    KEY idx_skill_ticket_messages_ticket (ticket_id, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci
  COMMENT='工单统一消息流,kind=text/image/file,body 为按 kind 变化的 JSON,取代原 comments+attachments';
```

down 迁移:`DROP TABLE skill_ticket_messages;` 并重建原 comments+attachments 两表(保证可回滚到结构,不恢复数据)。

**body JSON 形状(按 kind)**:
- `text`:`{"text": "<正文>"}`
- `image` / `file`:`{"object_path": "library/ticket-message/<...>", "file_name": "<原名>", "file_size": <字节>, "content_type": "<MIME>"}`
- `kind` 单独成列用于渲染分派/过滤;`image` 与 `file` 的区分在发送时按 `content_type`(`image/*` → image)决定;新增消息类型只需新 JSON 形状,不改表。

**Go 侧**:sqlc 将 JSON 列映射为 `json.RawMessage`;service 定义 per-kind 的 payload struct(`TextPayload{Text}`、`FilePayload{ObjectPath,FileName,FileSize,ContentType}`),按 kind marshal/unmarshal;DTO 给前端一个判别联合 `{ id, author_user_id, kind, created_at, ...payload }`。

**行为**:
- 提交工单:`title`/`description` 仍写 `skill_tickets`(作为「需求」,列表/详情头部用,**不算消息**);提交时上传的附件 → 落为该工单最早的 `image`/`file` 消息。
- 发消息:提交者与平台管理员都能发 `text`(输入框)与 `image`/`file`(上传按钮,每个文件一条消息)。
- **自动重开**:任意提交者消息(text/image/file)在 delivered/rejected 工单上触发 → pending(取代原「评论触发」)。
- 文件存储:复用 `LibraryBlobStore`,前缀改 `library/ticket-message/<ticket_id>/<uuid>`;下载走带鉴权的现有附件下载链路(归属校验:仅工单可见者可下载,沿用 `CanViewSkillTicket`)。
- 图片预览:前端按 `kind=image` 渲染缩略图(点开大图),下载同一鉴权链路取字节;`kind=file` 渲染文件名+大小+下载。

### 3. 可见范围编辑(交付后,新能力)

- 新增 service `UpdateCustomSkillTargets(ctx, principal, customSkillName, targets[])`:权限 `CanManageSkillTicket`;**覆盖写**——先 `DeleteCustomSkillTargetsByName(name)`(已存在),再逐条 `CreateCustomSkillTarget`。校验 audience ∈ {all_org, org_admins, requester_only}、org 存在、name 已交付。
- 新增 handler + 路由(平台侧):`PATCH /api/v1/skill-tickets/:id/targets`(工单中心,一工单一技能,handler 内由 ticket 取 custom_skill_name),请求体 `{ targets: [{org_id, audience}] }`。
- 前端:详情页 delivered 状态下「编辑可见范围」打开编辑器(复用交付弹窗里的 targets 编辑组件,抽成共享组件),保存调用上述接口。

### 4. 整页详情(双侧共享,role-aware)+ 列表 keep-alive

- **共享路由** `/skill-tickets/:id` → `TicketDetailPage.vue`,权限守卫 `CanViewSkillTicket`(platform_admin 或提交者本人)。平台队列与需求方列表都跳此路由。
- **role-aware**:
  - 平台管理员:显示 §1 情境化动作按钮(开始制作/交付/拒绝/重新受理/编辑可见范围)+ 报价编辑(pending/processing)。
  - 需求方(org_admin/org_member):**只读状态步骤条**,无管理员动作;交付后显示「去安装」(跳技能市场定制筛选并定位,沿用现有跳转);报价只读。
  - 双方都有:需求头部、可见范围展示、**对话(发消息:text/image/file)**。
- **页面结构**:顶部 header(标题 + `TicketStatusStepper` 状态步骤条 + 右侧动作区)→ 需求块(description)→ 可见范围块 → `TicketConversation`(消息流 + 输入/上传 composer)。
- **列表 keep-alive**:对两个列表所在路由做组件保活(平台队列页;需求方的 `/skills` 技能页),使「列表→详情→返回」保住滚动/筛选/激活 tab,不重新挂载;TanStack Query 缓存 + mutation 失效使列表行在返回时已反映最新状态。
- **共享组件**:`TicketStatusStepper.vue`、`TicketConversation.vue`(消息渲染 + composer,按 kind 渲染 text/image/file)、`TicketTargetsEditor.vue`(targets 编辑,交付弹窗与「编辑可见范围」共用)。

### 5. 平台队列与需求方列表(保留,仅瘦身)

- 平台 `CustomSkillTicketsPage.vue`:保留队列(筛选+表格),**移除**抽屉与其中的状态下拉/报价/拒绝/交付——这些迁入 `/skill-tickets/:id` 整页。表格「处理」改为跳整页。
- 需求方 `SkillTicketPanel.vue`:保留「我的工单」列表 + 提交表单,详情改为跳 `/skill-tickets/:id` 整页(去掉内嵌抽屉)。

## 后端改动面

- **迁移** `internal/migrations/000011_skill_ticket_messages.up/down.sql`(破坏性,见 §2)。
- **sqlc** `internal/store/queries/`:删 comments/attachments 相关 query,加 `skill_ticket_messages` 的 Create / ListByTicket;加 `UpdateCustomSkillTargets` 所需(已存在 Delete + Create)。
- **service** `skill_ticket_service.go`:`AddComment`→`SendMessage(kind, payload)`;`ListComments`→`ListMessages`;状态机方法重做为 §1 显式校验方法(StartProcessing/Reject/ReopenRejected/SetQuote 加前置校验);自动重开改为「提交者任意消息」触发;提交工单的附件落为消息。`custom_skill_service.go`:`Deliver` 加前置状态校验(rejected 禁止);新增 `UpdateCustomSkillTargets`。
- **authorizer**:沿用 `CanManageSkillTicket`/`CanViewSkillTicket`/`CanSubmitSkillTicket`;UpdateTargets 用 `CanManageSkillTicket`。如需新增谓词,扩展 `Can*`(遵循项目权限集中约定)。
- **handlers/router** `skill_tickets.go`、`custom_skills.go`:状态动作端点(开始制作/拒绝/重新受理 取代通用 status PATCH)、发消息(text)、上传消息(image/file,multipart)、列消息、更新 targets;移除旧 comment/attachment 端点。
- **OpenAPI 同步**:改 handler 后跑 `make openapi-gen` + `make web-types-gen`,提交保持同步(项目硬性要求)。
- **DTO**:`internal/api/handlers/dto.go` 请求体大写导出;响应用 `service.XxxResult`。

## 前端改动面

- **路由**:`web/src/app/router.ts` 加 `/skill-tickets/:id`(meta 不限角色;页面内按角色渲染);两个列表路由加 keep-alive(`DashboardLayout` 的 `<RouterView>` 包 `<keep-alive>`,按路由 name include)。
- **新页面**:`web/src/pages/skill-tickets/TicketDetailPage.vue`(role-aware)。
- **新共享组件**:`TicketStatusStepper.vue`、`TicketConversation.vue`、`TicketTargetsEditor.vue`(抽自现交付弹窗)。
- **改造**:`CustomSkillTicketsPage.vue`(队列瘦身,跳整页)、`SkillTicketPanel.vue`(列表+提交,跳整页)。
- **hooks** `useSkillTickets.ts`:消息 hooks(列消息、发 text、上传 image/file)、状态动作 hooks(开始制作/拒绝/重新受理/交付/更新 targets/设报价)、详情查询;mutation 成功失效列表/详情/角标/市场,保证 keep-alive 列表自动更新。
- **类型**:消息判别联合 `{ kind: 'text'|'image'|'file', ... }` 由 OpenAPI 生成或在 hooks 层封装。

## 数据流与错误处理

- 非法状态转换 → 后端 `ErrInvalid`(HTTP 400/409),前端 toast;按钮按当前状态条件渲染,从源头避免非法动作。
- 文件上传:复用现有大小/类型约束;`content_type` 决定 image/file kind;下载鉴权沿用 `CanViewSkillTicket` 归属校验(防 IDOR)。
- 迭代交付改名冲突 → 409,前端提示「迭代必须沿用同一技能名」(沿用现状)。
- targets 覆盖写在事务内(先删后插),失败回滚,避免可见范围半更新。

## 测试

- **后端单测**(testify):
  - 状态机:每个动作的合法转换 + 非法前置(如 delivered 不可拒绝、rejected 不可交付、pending 不可重新受理、delivered 不可改报价)逐一覆盖。
  - 消息:三种 kind 的创建与 body JSON 序列化/解析;提交工单时附件落为消息;提交者任意消息触发自动重开、管理员消息不触发。
  - targets 更新:覆盖写(增/删/改受众)、权限、未交付不可改、audience 校验。
  - 下载归属校验(非可见者 403)。
- **前端单测**(vitest):`TicketStatusStepper` 按状态渲染;`TicketConversation` 三种消息渲染 + composer 发送;`TicketDetailPage` role-aware(管理员见动作、需求方只读+去安装);keep-alive include 配置;hooks 失效正确。
- **真实浏览器三角色端到端**(遵循项目验证标准,逐项留证):
  - 平台管理员:队列→整页→开始制作→改报价→交付(自动已交付)→编辑可见范围→拒绝/重新受理;keep-alive 返回保位且行状态已更新。
  - 企业员工 & 企业管理员:提交(带图片/文件)→整页只读状态→发文字/图片/文件消息→在已交付/已拒绝发消息触发重开→交付后「去安装」。
  - 对话:图片内嵌预览、文件下载、时间戳、双向消息正确归属左右。
  - 差异化可见性经「编辑可见范围」调整后,市场可见性随之变化。

## 不做的事(YAGNI)

- 不迁移历史 comments/attachments 数据(破坏性迁移)。
- 不加状态 DB CHECK 约束(沿用应用层校验)。
- 不做「制作中→退回待处理」、不做手动设已交付。
- 不引入支付(报价仍仅展示字段)。
- 消息不做编辑/撤回/已读回执;不做富文本(text 为纯文本)。

## 影响面小结

- 数据库:破坏性迁移 000011(删 2 表、建 1 表)。
- 后端:状态机校验收紧、消息模型重构、targets 更新接口、相关 handler/route/sqlc/OpenAPI。
- 前端:新增共享整页详情路由 + 三个共享组件 + keep-alive;平台队列与需求方面板瘦身为「列表→整页」。
- 风险:破坏性迁移(已确认无需兼容);双侧 UI 重构面较大,靠单测 + 三角色浏览器回归覆盖。
