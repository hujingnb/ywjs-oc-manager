# 单元测试场景注释补齐设计

## 背景

仓库规范要求新增或重构代码具备必要中文注释。当前需要补充一条更明确的测试规范：每一个单元测试方法或子测试逻辑，都必须用中文注释说明该测试覆盖的业务场景、边界条件或异常路径。

现有 Go 测试中已经有部分文件级或测试级说明，但仍有大量 `func Test...` 和 `t.Run(...)` 缺少相邻的场景注释。本次变更要让当前项目也符合新规则，而不是只约束后续新增测试。

## 目标

- 在根目录 `AGENTS.md` 的“单元测试”章节新增明确规则。
- 为当前所有 `*_test.go` 中缺少说明的顶层测试方法补充中文注释。
- 为当前所有缺少说明的 `t.Run(...)` 子测试逻辑补充中文注释。
- 仅添加说明性注释，不修改测试逻辑、断言、测试数据或执行路径。

## 非目标

- 不重构测试结构。
- 不调整测试命名。
- 不替换断言库或改写已有断言。
- 不新增强制检查脚本，除非后续另有明确需求。

## 注释规则

- 顶层测试方法：每个 `func TestXxx(t *testing.T)` 前应有相邻中文注释，说明该测试验证什么场景。
- 子测试逻辑：每个 `t.Run(...)` 前应有相邻中文注释，说明该子场景覆盖的输入、边界或异常路径。
- 文件级 `Package ...` 注释不能替代单个测试方法或子测试场景注释。
- 注释应说明测试意图，避免只重复函数名或代码表面行为。

## 实施方案

采用一次性补齐全仓库测试注释的方案。

1. 更新 `AGENTS.md` 的“单元测试”章节，新增测试方法和子测试场景注释规则。
2. 扫描所有 `*_test.go`，定位缺少相邻注释的 `func Test...` 和 `t.Run(...)`。
3. 按测试名称、断言和上下文补充中文注释。
4. 保持改动聚焦，不改变测试逻辑或无关格式。

## 验证

- 运行静态扫描，确认所有 `func Test...` 和 `t.Run(...)` 前都有相邻注释。
- 运行 `go test ./...` 验证测试行为未被注释改动破坏。
- 若全量测试因外部依赖或环境变量无法完成，在交付说明中明确失败原因和风险。

## 风险

- 主要风险是批量注释造成较大 diff，影响评审效率。
- 由于不改测试逻辑，行为风险较低。
- 部分测试名称较抽象时，需要结合断言和上下文写注释，避免注释误导后续维护者。
